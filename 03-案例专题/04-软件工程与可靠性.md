# 专题四：软件工程与可靠性

> 来源：系统架构设计师知识大纲 + Gemini 对话补完 + 历年真题考点

---

## 一、核心知识体系

### 1.1 软件可靠性 vs 可用性（易混淆）

| 维度 | 可靠性 (Reliability) | 可用性 (Availability) |
|------|---------------------|----------------------|
| **定义** | 系统在规定时间内、规定条件下完成规定功能的能力 | 系统能够正常运行的时间比例 |
| **度量指标** | **MTBF（Mean Time Between Failures，平均无故障时间）** | **MTTR（Mean Time To Repair，平均修复时间）** |
| **公式** | MTBF = 总运行时间 / 故障次数 | 可用性 = MTBF / (MTBF + MTTR) |
| **大白话** | 能不能一直不出错？ | 坏了能不能快点修好？ |
| **提升策略** | 冗余设计、容错、错误检测 | 故障自动转移、快速恢复机制 |

**可用性级别（几个 9）：**
- 99.9% = 3个9 ≈ 每年停机 8.76 小时
- 99.99% = 4个9 ≈ 每年停机 52.6 分钟
- 99.999% = 5个9 ≈ 每年停机 5.26 分钟

### 1.2 可靠性计算

| 系统类型 | 计算方式 | 公式 |
|---------|---------|------|
| **串联系统** | 所有组件都必须正常 | R = R₁ × R₂ × ... × Rn |
| **并联系统** | 只要有一个组件正常即可 | R = 1 - (1-R₁)(1-R₂)...(1-Rn) |
| **混联系统** | 先算并联，再算串联 | 分解为子模块分别计算 |

**要点**：串联系统可靠性比任一组件都低；并联系统可靠性高于任一组件。

### 1.3 软件开发模型

| 开发模型 | 特点 | 适用场景 | 核心驱动 |
|---------|------|---------|---------|
| **瀑布模型** | 阶段分明、文档驱动、不可逆 | 需求明确、变更少 | 文档驱动 |
| **原型模型** | 快速构建原型，用户反馈迭代 | 需求不明确 | 用户反馈 |
| **螺旋模型** | 原型 + 瀑布 + **风险分析** | 大型复杂、高风险项目 | **风险驱动**（常考） |
| **敏捷开发** | 迭代增量、用户参与、拥抱变化 | 需求变化快 | 价值驱动 |
| **增量模型** | 分批次交付功能 | 需要快速交付部分功能 | 功能驱动 |
| **V 模型** | 每个开发阶段对应测试阶段 | 强调测试的严肃项目 | 测试驱动 |

**螺旋模型核心**：每一轮迭代都包含四个象限：目标设定 → 风险分析 → 开发验证 → 计划下一轮。**风险驱动**是关键词。

### 1.4 遗留系统演进策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **逆向工程** | 从代码反推设计文档和架构理解 | 文档缺失、代码重复率高的旧系统 |
| **代码重构** | 不改变外部行为的前提下改善内部结构 | 降低代码重复率、提高可维护性 |
| **再工程** | 重新设计和重新实现系统 | 旧系统无法维护、需要彻底改造 |
| **绞杀者模式 (Strangler Fig)** | 在旧系统外围开发新功能，逐步替换旧模块 | 高风险的大型系统迁移（常考） |

**绞杀者模式深入（实战神技）：**
- **核心思想**：不要试图一次性重写
- **做法**：新需求在新架构上做，拦截旧系统入口
- **结果**：新服务逐渐替代旧逻辑，旧系统自然"枯萎"
- **对你提到的"重复率高、文档缺失"系统尤为重要**

### 1.5 软件维护（四类）

| 维护类型 | 说明 | 占比 | 考点 |
|---------|------|------|------|
| **完善性维护** | 增加新功能、提升性能 | **约 50%（最高）** | 🔴 最常考 |
| **适应性维护** | 适应新环境（新 OS、新数据库） | 约 25% | 🟡 次常考 |
| **正确性维护** | 修复缺陷和 Bug | 约 20% | 🟡 |
| **预防性维护** | 为提高可维护性而提前修改 | 约 5% | 🟢 少考 |

### 1.6 需求工程

**需求层次（自顶向下）：**
业务需求 → 用户需求 → 功能需求 → 非功能需求

**需求获取技术（常考排序/匹配）：**
- **访谈**：面对面交流，适合少数关键用户
- **问卷调查**：大规模收集意见，适合广泛调查
- **头脑风暴**：激发创意，适合探索性需求
- **原型法**：快速原型展示，适合需求不明确
- **JAD（Joint Application Development，联合应用开发）**：集中会议讨论，适合多方协作
- **观察法**：现场观察用户操作，适合流程优化
- **文档分析**：分析现有文档，适合遗留系统

### 1.7 DFD（Data Flow Diagram，数据流图）

**DFD 三层结构：**
- **顶层（上下文图）**：系统边界，展示系统与外部实体的交互
- **0 层图**：主要加工和数据存储，展示系统内部核心功能
- **1 层图**：对 0 层图中的加工进一步细化

**数据平衡原则**：加工的输入必须能产生所有输出（父图与子图平衡）

**DFD 三类错误：**
1. 外部实体之间直接通信（必须通过加工中转）
2. 外部实体直接访问数据存储（必须通过加工）
3. 数据存储之间直接传递数据（必须通过加工）

**DFD vs 系统流程图的区别：**
- DFD 关注**数据流和功能**（逻辑模型），不涉及物理设备和控制流
- 系统流程图关注**物理组件和控制流**（物理模型），包含硬件设备和操作顺序

**数据字典组成**：
- 数据项、数据结构、数据流、数据存储、处理逻辑、外部实体
- 数据字典是对 DFD 中所有元素的精确描述和定义

### 1.8 UML（Unified Modeling Language，统一建模语言）建模

| 图类型 | 用途 | 包含要素 |
|--------|------|---------|
| **用例图** | 功能需求建模 | 参与者、用例、关系（包含/扩展/泛化） |
| **类图** | 静态结构建模 | 类、接口、关系（关联/聚合/组合/继承） |
| **序列图** | 交互行为建模 | 对象、消息（同步/异步）、生命线、激活期 |
| **状态图** | 对象状态变化 | 状态、转换、事件、动作 |
| **活动图** | 业务流程建模 | 活动、决策、并发、泳道 |

**面向对象三模型：**
- **对象模型**：静态结构，描述类、对象及其关系（类图）
- **动态模型**：状态变化和时序行为（状态图、序列图）
- **功能模型**：数据变换和功能需求（用例图、DFD）

## 三、案例分析答题模板

### DFD 数据流图补全题答题思路

```
1. 确认 DFD 三层结构：顶层（系统边界）、0层（主要加工）、1层（详细加工）
2. 数据平衡原则：加工的所有输入必须能产生所有输出
3. 检查三类错误：
   - 外部实体之间直接通信（必须通过加工）
   - 外部实体直接访问数据存储（必须通过加工）
   - 数据存储之间直接传递（必须通过加工）
4. 结合案例描述补充缺失的外部实体、加工或数据流
```

### 遗留系统迁移题答题模板

```
遗留系统改造方案：
1. 现状分析：代码重复率高、文档缺失、技术栈老旧
2. 逆向工程：梳理现有业务逻辑，补充架构文档
3. 渐进式迁移（绞杀者模式）：
   - 新需求在新架构上实现
   - 使用 API 网关拦截旧系统入口
   - 逐步将旧功能迁移至新服务
4. 代码重构：SonarQube 扫描，降低重复率，拆分复杂函数
5. 数据迁移：双写 + CDC 同步，确保平滑过渡
```

## 四、知识卡片·速记

| 概念 | 关键词 | 一句话记忆 |
|------|--------|-----------|
| MTBF | 平均无故障时间 | 多久不出错 |
| MTTR | 平均修复时间 | 坏了多快修好 |
| 串联系统 | 全都要对 | R = 连乘，越串越不可靠 |
| 并联系统 | 有一个对就行 | R = 1-连乘失败，越并越可靠 |
| 螺旋模型 | 风险分析 | 每一轮都要评估风险 |
| 完善性维护 | 加功能 | 占维护工作量一半 |
| 适应性维护 | 换环境 | 适应新 OS/新数据库 |
| 正确性维护 | 修 Bug | 修复缺陷 |
| 预防性维护 | 提前改善 | 为将来做准备 |
| 逆向工程 | 代码→设计 | 从实现反推文档 |
| 绞杀者模式 | 渐进替换 | 藤蔓缠树，逐步替换 |
| 重构 | 内部改善 | 不改行为，改善结构 |
| 再工程 | 重新设计 | 推倒重来 |
| DFD 数据平衡 | 输入产输出 | 加工不能凭空产出 |
| 外部实体 | 系统外的人/物 | DFD 的最外圈 |
