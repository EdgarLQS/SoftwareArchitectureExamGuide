## 一、核心知识体系

### 1.1 设计模式（23 种分类）

设计模式在综合知识选择题中占 **5-8 分**。架构师不需要背诵全部，但必须能根据场景分类。

#### 创建型模式（解决"怎么造对象"）

| 模式 | 说明 | 必考场景 |
|------|------|---------|
| **单例 (Singleton)** | 全局只有一个实例 | 配置管理器、数据库连接池 |
| **工厂 (Factory)** | 用工厂方法创建对象 | 需要灵活创建不同类型对象 |
| **原型 (Prototype)** | 通过克隆创建对象 | 创建成本高的对象复制 |
| 建造者 (Builder) | 分步骤构建复杂对象 | 构建复杂对象（如房屋） |
| 抽象工厂 (Abstract Factory) | 创建一系列相关对象 | 跨平台 UI 组件族 |

#### 结构型模式（解决"怎么拼对象"）

| 模式 | 说明 | 必考场景 |
|------|------|---------|
| **适配器 (Adapter)** | 接口不兼容时使用 | 接旧系统常用、数据格式转换 |
| **装饰器 (Decorator)** | 不修改原代码的情况下动态添加功能 | 不改代码加功能 |
| **外观 (Facade)** | 为子系统提供统一接口 | 微服务网关原型、简化接口 |
| 代理 (Proxy) | 控制对对象的访问 | 远程代理、虚拟代理、保护代理 |
| 桥接 (Bridge) | 将抽象与实现分离 | 多维度变化解耦 |
| 组合 (Composite) | 树形结构表示整体-部分关系 | 文件系统、菜单 |
| 享元 (Flyweight) | 共享细粒度对象节省内存 | 文字编辑器字符对象 |

**架构师视角**：微服务中的"断路器"其实是**代理模式**的一种演进。

#### 行为型模式（解决"对象怎么沟通"）

| 模式 | 说明 | 必考场景 |
|------|------|---------|
| **策略 (Strategy)** | 定义算法族，运行时切换 | 不同折扣算法切换、合规规则切换 |
| **观察者 (Observer)** | 一对多依赖，状态改变通知所有依赖 | 消息队列原型、事件通知 |
| **状态 (State)** | 对象状态不同行为不同 | 订单状态流转（待支付→已支付→发货） |
| 模板方法 (Template Method) | 定义算法骨架，子类实现步骤 | 框架生命周期 |
| 命令 (Command) | 将请求封装为对象 | 撤销/重做、任务队列 |
| 迭代器 (Iterator) | 顺序访问集合元素 | 遍历集合 |
| 责任链 (Chain of Responsibility) | 请求沿处理链传递 | 审批流程、过滤器链 |
| 中介者 (Mediator) | 通过中介者解耦多对象交互 | 聊天室、航空管制 |
| 备忘录 (Memento) | 保存对象状态以便恢复 | 撤销操作 |
| 访问者 (Visitor) | 不改变数据结构的前提下定义新操作 | 报表生成 |
| 解释器 (Interpreter) | 定义语言语法并解释执行 | 规则引擎 |

#### 策略模式进阶案例

场景：全球支付合规校验，需要灵活切换不同国家的合规规则（GDPR（General Data Protection Regulation，通用数据保护条例）、中国数据安全法等）。

**实现**：定义 `ComplianceStrategy` 接口，实现 `GDPRStrategy`、`ChinaDataLawStrategy` 等。
**架构收益**：实现了**开闭原则 (Open-Closed Principle)** —— 新增合规要求只需增加新策略类，不修改支付主流程代码。

### 1.2 信息安全架构

#### 加密体系

| 类型 | 代表算法 | 特点 | 应用场景 |
|------|---------|------|---------|
| **对称加密** | AES（Advanced Encryption Standard，高级加密标准）、DES（Data Encryption Standard，数据加密标准）、国密 SM1 | 加密/解密用同一把密钥，速度快 | 大数据量加密 |
| **非对称加密** | RSA（Rivest-Shamir-Adleman）、国密 SM2 | 公钥加密、私钥解密，速度慢但安全 | 密钥交换、数字签名 |

#### 访问控制模型

| 模型 | 说明 | 特点 | 考点 |
|------|------|------|------|
| **DAC（Discretionary Access Control，自主存取控制）** | 资源主人说了算 | 灵活性高，安全性较低 | Unix 文件权限 |
| **MAC（Mandatory Access Control，强制存取控制）** | 系统安全策略说了算 | 军事级、保密级 | 安全级别标签 |
| **RBAC（Role-Based Access Control，基于角色的访问控制）** | 用户→角色→权限 | **最常考** | 企业系统标准 |

#### Web 安全

| 机制 | 说明 | 防护方式 |
|------|------|---------|
| **OAuth 2.0（Open Authorization，开放授权）** | 授权框架，第三方应用获取用户资源 | 令牌机制 |
| **JWT（JSON Web Token）** | 自包含的身份令牌 | 无状态认证 |
| **CORS（Cross-Origin Resource Sharing，跨域资源共享）** | 跨域资源共享 | 预检请求、Access-Control 头 |
| **数字签名** | 使用私钥签名、公钥验证 | 确保数据完整性和来源可信 |
| **SSL/TLS（Secure Sockets Layer/Transport Layer Security，安全套接层/传输层安全协议）** | 安全传输协议 | HTTPS（HTTP Secure）底层协议 |

### 1.3 法律法规（送分题）

| 考点 | 内容 | 备注 |
|------|------|------|
| **职务作品归属** | 利用公司设备、为完成公司任务写的代码，著作权归**公司** | 常考 |
| **软件著作权产生时间** | 从软件**开发完成之日**起自动产生，不需要等登记 | 登记只是为了证据 |
| **侵权判定** | 功能一样但代码实现完全不同，通常**不构成**著作权侵权 | 著作权保护"表达"不保护"思想" |
| **等保 2.0** | 网络安全等级保护制度，分五个等级 | 合规底线 |
| **国家标准** | GB/T（GuoBiao/TuiJian，国家推荐标准）8567（软件文档规范）、GB/T 11457（软件工程术语） | 选择题可能涉及 |

### 1.3.1 信息系统安全保障措施

| 措施 | 说明 |
|------|------|
| **身份认证** | 密码、生物识别、双因素认证（2FA） |
| **访问控制** | RBAC、MAC、DAC 模型 |
| **加密** | 对称/非对称加密、哈希算法 |
| **审计** | 日志记录、行为追踪、合规报告 |
| **入侵检测（IDS，Intrusion Detection System）** | 实时监控网络异常 |
| **防火墙** | 包过滤、状态检测、应用层过滤 |

### 1.4 企业应用集成（EAI，Enterprise Application Integration）四个层次

| 层次 | 说明 | 技术 |
|------|------|------|
| **数据级集成** | 数据同步和共享，不同系统共用数据 | ETL（Extract-Transform-Load）、数据复制 |
| **接口级集成** | 通过 API 实现系统间功能调用 | Web Service（SOAP/WSDL/UDDI）、RESTful API |
| **业务流程级集成** | 跨系统的业务流程协调和编排 | BPM（Business Process Management）、工作流引擎 |
| **界面级集成** | 多个系统界面整合为统一入口 | 企业门户、SSO（Single Sign-On，单点登录） |

### 1.5 区块链六层架构与智能合约

| 层级 | 说明 | 关键技术 |
|------|------|---------|
| **数据层** | 区块链的数据结构 | 区块、链式结构、Merkle 树、哈希算法 |
| **网络层** | 节点之间的通信机制 | P2P 网络、传播协议 |
| **共识层** | 节点达成一致 | PoW（工作量证明）、PoS（权益证明）、PBFT |
| **激励层** | 经济激励机制 | 发行机制、分配机制 |
| **合约层** | 可编程的逻辑 | 智能合约（自动执行的代码）、脚本 |
| **应用层** | 基于区块链的应用 | DApp、DeFi、供应链管理 |

**智能合约**：部署在区块链上的程序，当满足预设条件时自动执行，不可篡改、不可逆转。2025 年上真题考查区块链六层架构 + 智能合约在农产品溯源中的应用。

### 1.7 危险驱动安全分析

- **概念**：从系统可能面临的危险/威胁出发，反向推导安全需求
- **流程**：危险识别 → 风险评估 → 安全需求定义 → 安全架构设计 → 验证
- **2024 年下新增热点**：胰岛素泵等医疗设备的安全分析

### 1.8 PACELC（Partition And Consistency Else Latency And Consistency）定理深入

CAP 定理的扩展版本。CAP 只考虑了网络分区 (P) 时的权衡，PACELC 补充了正常情况：

- **P（Partition，分区发生时）**：在一致性 (C) 和可用性 (A) 之间选择
- **EL（Else，正常情况）**：在延迟 (L) 和一致性 (C) 之间选择

**场景**：跨境电商在全球访问时，即使网络正常（没有分区），也需要在**降低访问延迟**和**保持数据一致性**之间权衡。解决方案包括边缘计算 (Edge Computing)、CDN 缓存、多地域部署等。

### 1.5 边缘计算 (Edge Computing)

| 维度 | 说明 |
|------|------|
| **核心思想** | 将计算和存储靠近数据源，减少数据传输到云端的延迟 |
| **优势** | 低延迟、减少带宽消耗、支持离线运行 |
| **场景** | IoT（Internet of Things，物联网）设备实时处理、CDN（Content Delivery Network，内容分发网络）内容分发、自动驾驶 |
| **与云计算的关系** | 互补而非替代，边缘处理实时，云端处理批量和存储 |

## 三、答题模板

### 设计模式选择题答题思路

```
1. 识别题目中问题的本质：
   - "创建对象" → 创建型模式
   - "组合/适配对象" → 结构型模式
   - "对象间通信/行为切换" → 行为型模式
2. 根据具体场景进一步缩小范围
3. 排除法：不属于该类型的模式直接排除
```

### 安全架构答题模板

```
安全方案设计：
1. 身份认证：OAuth 2.0 + JWT 实现无状态认证
2. 访问控制：RBAC 模型，用户→角色→权限三层设计
3. 数据安全：传输层 SSL/TLS，存储层 AES 加密
4. 数字签名：非对称加密（RSA/SM2）确保数据完整性
5. 等保合规：按照等保 2.0 要求进行安全评估
```

## 五、嵌入式系统补充（专题五考点）

### 5.1 嵌入式系统特点
- **专用性**：面向特定应用场景
- **实时性**：需要在规定时间内响应
- **资源受限**：CPU、内存、存储等资源有限
- **可靠性要求高**：通常用于关键任务场景
- **功耗敏感**：电池供电设备需优化功耗

### 5.2 实时调度算法

| 算法 | 优先级策略 | 特点 |
|------|-----------|------|
| **RMS（Rate Monotonic Scheduling，速率单调调度）** | 周期越短优先级越高 | 静态优先级、可抢占 |
| **EDF（Earliest Deadline First，最早截止期限优先）** | 截止期限越近优先级越高 | 动态优先级、可抢占 |

### 5.3 FACE（Future Airborne Capability Environment，未来机载能力环境）架构五段

| 段 | 说明 |
|---|------|
| **I/O Services** | 输入输出服务，硬件抽象层 |
| **Core Services** | 核心服务，操作系统基础功能 |
| **Platform-Specific Services** | 平台特定服务 |
| **OSEK/VDX** | 标准化操作系统接口 |
| **Application** | 应用层，业务逻辑 |

**可移植性**：FACE 的核心目标是实现应用软件在不同硬件平台间的可移植。

### 5.4 CPS（Cyber-Physical System，信息物理系统）三层架构

| 层级 | 说明 | 安全威胁 |
|------|------|---------|
| **感知层** | 传感器、执行器采集数据 | 传感器欺骗、数据篡改 |
| **网络层** | 数据传输和通信 | 中间人攻击、数据窃听 |
| **应用层** | 数据处理和业务逻辑 | 未授权访问、拒绝服务 |

### 5.5 ROS（Robot Operating System，机器人操作系统）

**四种通信机制：**
- **Topic（话题）**：发布-订阅模式，适合传感器数据流
- **Service（服务）**：请求-响应模式，适合一次性调用
- **Action（动作）**：长时间运行的异步任务，支持反馈和取消
- **Parameter（参数）**：全局配置参数

**三层架构**：文件系统层 → 计算图层 → 社区层

### 5.6 智能化操作系统（AI for OS / OS for AI）

- **AI for OS**：利用 AI 优化操作系统运行（智能调度、故障预测）
- **OS for AI**：操作系统为 AI 计算提供底层支持（GPU 管理、分布式训练调度）
- **2025 年下新增热点**

### 5.7 轻量化内核

- 面向 IoT 和嵌入式设备的小型化操作系统内核
- 典型代表：FreeRTOS、Zephyr、LiteOS
- 特点：内存占用小（KB 级别）、功耗低、实时性好

### 5.8 IoT（Internet of Things，物联网）架构对比

| 架构 | 特点 | 适用场景 |
|------|------|---------|
| **家庭网关架构** | 本地处理，低延迟，隐私保护 | 智能家居、本地控制 |
| **云平台架构** | 集中管理，大数据分析 | 城市物联网、工业 IoT |

### 5.9 MQTT（Message Queuing Telemetry Transport，消息队列遥测传输协议）

- **模式**：发布-订阅，轻量级
- **QoS（Quality of Service，服务质量）三个级别**：
  - QoS 0：至多一次（可能丢失）
  - QoS 1：至少一次（可能重复）
  - QoS 2：恰好一次（不丢失不重复）

### 5.10 TCP vs UDP 协议对比

| 特性 | TCP（Transmission Control Protocol，传输控制协议） | UDP（User Datagram Protocol，用户数据报协议） |
|------|-------------------------------------|-------------------------------|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠（确认、重传、排序） | 不可靠 |
| 速度 | 慢（额外开销） | 快 |
| 场景 | 文件传输、邮件、Web | 视频流、在线游戏、DNS |
