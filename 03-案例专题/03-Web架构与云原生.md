## 一、核心知识体系

### 1.1 微服务架构

**Web 架构演进路径：**
单体架构 → 垂直拆分 → SOA 架构 → 微服务架构 → 云原生架构

**微服务核心特征（7 条）：**
1. 服务组件化（独立部署）
2. 按业务组织（非技术分层）
3. 智能端点、哑管道（服务间直接通信）
4. 去中心化治理（各服务可选不同技术栈）
5. 去中心化数据管理（每个服务有自己的数据库）
6. 基础设施自动化（CI/CD、自动部署）
7. 容错设计（熔断、降级、限流）

**核心组件（必背）：**

| 组件 | 代表技术 | 作用 |
|------|---------|------|
| 服务发现 | Eureka、Consul、Nacos | 解决"谁在哪里"的问题 |
| API（Application Programming Interface，应用程序编程接口）网关 | Zuul、Spring Cloud Gateway | 路由、鉴权、限流、统一入口 |
| 配置中心 | Apollo、Nacos | 环境配置动态切换 |
| 断路器 | Hystrix、Sentinel | 防止雪崩效应 |

**拆分原则：** 单一职责、关注点分离、围绕业务建模

### 1.2 SOA（Service-Oriented Architecture，面向服务架构）深入

**SOA 三角色：**
- **服务提供者（Service Provider）**：发布服务、实现业务逻辑、接收请求
- **服务消费者（Service Consumer）**：发现并调用服务
- **服务注册中心（Service Registry）**：服务的目录，提供服务发现功能

**Web Service 技术栈：**
- **SOAP（Simple Object Access Protocol，简单对象访问协议）**：基于 XML 的消息传输协议
- **WSDL（Web Services Description Language，Web 服务描述语言）**：XML 格式描述服务的接口、参数、返回值
- **UDDI（Universal Description, Discovery and Integration，统一描述、发现和集成）**：服务注册和发现的协议
- **BPEL（Business Process Execution Language，业务流程执行语言）**：基于 Web Service 的业务流程编排语言

**ESB（Enterprise Service Bus，企业服务总线）：**
- **定义**：中间件基础设施，提供服务的通信、转换和编排
- **核心功能**：协议转换、消息路由、数据格式转换、服务编排
- **适用场景**：异构系统集成、企业级服务复用

### 1.3 微服务治理三板斧

| 机制 | 核心思想 | 类比 | 场景 |
|------|---------|------|------|
| **熔断 (Circuit Breaker)** | A 调 B，B 坏了，A 主动断开不再请求 B | 电路跳闸 | 依赖服务持续超时/报错 |
| **降级 (Fallback)** | B 坏了，A 不报错，返回默认值 | 备用电源 | 非核心功能不可用时 |
| **限流 (Rate Limiting)** | 流量超过阈值，直接拒绝多余请求 | 限号通行 | 大促/秒杀保护系统不崩溃 |

**其他治理要点：**
- **服务发现与注册中心**：服务启动时登记，调用方拉取名单
- **灰度/金丝雀发布**：10% 流量新版本，90% 旧版本，逐步切换
- **全链路追踪**：Skywalking、Zipkin，通过 TraceID 定位性能瓶颈

### 1.3 负载均衡

| 层次 | 依据 | 代表技术 | 特点 |
|------|------|---------|------|
| **L4 传输层** | IP + 端口 | LVS、F5 | 性能极高，不感知业务 |
| **L7 应用层** | HTTP（HyperText Transfer Protocol，超文本传输协议）、URL、Cookie | Nginx、HAProxy | 复杂业务路由 |
| **DNS（Domain Name System，域名系统）负载均衡** | 域名解析 | 各 DNS 服务商 | 跨国/跨地域流量分发 |

### 1.4 容器化与 Kubernetes 编排

**K8s 核心对象：**

| 对象 | 说明 | 考点 |
|------|------|------|
| **Pod** | K8s 最小调度单位，一组共享网络和存储的容器 | 为什么要有 Pod？解决容器间强耦合（如业务+日志 Sidecar） |
| **Deployment** | 定义副本数量，保证始终有 N 个副本运行 | 无状态应用的部署管理 |
| **Service** | 提供稳定的虚拟 IP（Pod IP 会变） | ClusterIP（内部）、NodePort（外部）、LoadBalancer（云平台） |
| **Ingress** | K8s 的 Nginx，处理域名、路径转发和 SSL 卸载 | 外部流量进入集群的入口 |
| **ConfigMap/Secrets** | 配置与代码解耦 | 环境配置管理 |
| **Namespace** | 逻辑隔离 | 开发/测试/生产环境隔离 |

**关键能力：**

| 能力 | 说明 | 场景 |
|------|------|------|
| **HPA（Horizontal Pod Autoscaler，水平 Pod 自动扩缩容）** | 监控 CPU/内存利用率，超阈值自动增加副本数 | 大促自动应对流量高峰 |
| **健康检查** | Liveness Probe（存活探针）、Readiness Probe（就绪探针） | 故障自动恢复 |
| **CA（Cluster Autoscaler，集群自动扩缩容）** | 底层机器不足时自动增加节点 | 配合 HPA 实现分钟级扩容 |

### 1.5 服务网格 (Service Mesh)

| 维度 | 说明 |
|------|------|
| **核心理念** | 将服务间通信逻辑（限流、加密、重试）从业务代码剥离，下沉到基础设施层 |
| **代表技术** | Istio |
| **Sidecar 模式** | 边车模式，负责处理网络流量而不侵入业务代码 |
| **架构优势** | 对业务逻辑**无侵入**，适合旧系统现代化改造 |

### 1.6 Web 安全

| 安全机制 | 说明 | 考点 |
|---------|------|------|
| **OAuth 2.0（Open Authorization，开放授权）+ JWT（JSON Web Token）** | 微服务下的授权和身份校验标准 | 无状态认证 |
| **CORS（Cross-Origin Resource Sharing，跨域资源共享）** | Web 前后端分离后的经典问题 | 预检请求、Access-Control |
| **SQL 注入** | 通过拼接 SQL 执行恶意代码 | 参数化查询、预编译 |
| **XSS（Cross-Site Scripting，跨站脚本攻击）** | 在页面注入恶意脚本 | 输入过滤、输出编码 |
| **CSRF（Cross-Site Request Forgery，跨站请求伪造）** | 冒充用户发起请求 | Token 验证、SameSite Cookie |

### 1.7 DDD（Domain-Driven Design，领域驱动设计）

**核心概念：**
- **限界上下文（Bounded Context）**：明确的业务边界，每个上下文有自己的模型和语言
- **聚合根（Aggregate Root）**：聚合的入口，外部只能通过聚合根访问内部对象
- **实体（Entity）**：有唯一标识、有生命周期的对象
- **值对象（Value Object）**：无标识、不可变的对象
- **通用语言（Ubiquitous Language）**：业务人员和开发人员统一使用的术语

**上下文映射关系：**
- 合作关系（Partnership）、客户-供应商（Customer-Supplier）、遵奉者（Conformist）
- 防腐层（ACL，Anti-Corruption Layer）：防止外部模型污染内部模型

**2025 年下真题**：考查 DDD 限界上下文在弱网/断网场景下的应用。

### 1.8 弱网/断网处理与数据最终一致性

**挑战**：移动端在网络不稳定时容易出现请求超时、数据不一致
**方案**：
- **离线缓存**：本地存储关键数据，网络恢复后同步
- **乐观锁 + 版本号**：允许多端并发修改，通过版本号解决冲突
- **补偿机制**：失败操作记录重试队列，自动补偿
- **幂等设计**：确保重复请求不会产生副作用

### 1.9 中台架构

| 中台类型 | 核心内容 | 代表组件 |
|---------|---------|---------|
| **业务中台** | 可复用的业务能力 | 用户中心、订单中心、支付中心 |
| **数据中台** | 统一数据资产和服务 | 数据仓库、数据湖、数据分析平台 |
| **技术中台** | 通用技术能力 | 微服务框架、DevOps 平台、网关服务 |

**核心价值**：避免重复造轮子，快速支撑前台业务创新。

## 三、微服务治理 vs K8s 编排对照表

| 维度 | 微服务治理（软件层面） | K8s 编排（基础设施层面） |
|------|----------------------|----------------------|
| **负载均衡** | Ribbon / Spring Cloud LoadBalancer | Service / Kube-proxy / Ingress |
| **高可用** | 重试、熔断、降级 | 副本集 (ReplicaSet)、健康检查 (Liveness Probe) |
| **伸缩性** | 动态调整线程池、异步化 | HPA（水平扩缩容）、CA（集群扩缩容） |
| **流量管理** | 灰度发布、金丝雀发布 | Sidecar (Istio)、网关转发 |
| **配置管理** | 配置中心（Nacos / Apollo） | ConfigMap / Secrets |

## 四、案例分析答题模板

### 微服务迁移题答题模板

```
单体转微服务方案：
1. 服务拆分：按业务领域拆分（单一职责），每个服务独立数据库
2. 服务发现：引入 Nacos/Consul 实现服务自动注册与发现
3. API 网关：引入 Spring Cloud Gateway 统一入口，处理路由、鉴权、限流
4. 配置管理：使用 Apollo/Nacos 实现配置动态刷新
5. 容错设计：关键链路配置熔断、降级、限流策略
6. 全链路监控：引入 Skywalking/Zipkin 追踪调用链
```

### 云原生部署题答题模板

```
云原生迁移方案：
1. 容器化：编写 Dockerfile，将应用打包为容器镜像
2. K8s 部署：定义 Deployment（副本数）+ Service（稳定访问入口）
3. 弹性扩容：开启 HPA，根据 CPU/内存自动扩缩容
4. 配置解耦：使用 Namespace 隔离环境，ConfigMap 管理配置
5. 灰度发布：引入 Istio，通过 VirtualService 权重实现金丝雀发布
6. 日志监控：统一日志收集（EFK）+ 全链路追踪（Skywalking）
```

### 高并发场景解决方案

```
场景：大促期间服务调用链太深，前端响应变慢
方案：
1. 引入服务网格（Istio），配置请求超时和重试机制，开启熔断逻辑
2. 异步化改造：非核心链路（发优惠券、发邮件）通过 MQ 异步化
3. 部署全链路监控（Skywalking）定位具体性能瓶颈
4. 针对热点服务开启 HPA 自动扩容
```
