# 案例分析专题复习指南

## 一、考试概述

### 1.1 考试形式
- **考试时间**：150 分钟（2.5 小时）
- **题型**：4 道大题选答 3 题，每题 25 分，共 75 分
- **及格线**：45 分
- **答题方式**：机考（打字作答）

### 1.2 题型特点
- **试题一**：必答题（25 分）- 通常考查架构设计/质量属性
- **试题二 ~ 四**：选答题（25 分/题）- 按专题分类

## 二、六大专题知识体系

### 专题一：系统架构设计（高频必考）

#### 2.1.1 架构风格分类

| 架构风格 | 特点 | 适用场景 |
|---------|------|---------|
| **分层架构** | 层次分明、职责分离 | 企业应用系统 |
| **C/S 架构** | 响应快、交互强 | 桌面应用、实时系统 |
| **B/S 架构** | 零客户端、易维护 | Web 应用、互联网应用 |
| **微服务架构** | 独立部署、服务自治 | 大型复杂系统 |
| **SOA 架构** | 服务复用、ESB 总线 | 企业集成 |
| **管道 - 过滤器** | 数据流处理 | 编译器、数据处理 |
| **事件驱动架构** | 松耦合、异步处理 | 消息系统、实时通知 |
| **微内核架构** | 核心 + 插件 | IDE、可扩展系统 |

#### 2.1.2 质量属性（重点掌握）

**六大质量属性定义与实现策略：**

1. **性能（Performance）**
   - 定义：系统响应时间和吞吐量
   - 实现策略：缓存、异步处理、负载均衡、数据分片

2. **可靠性（Reliability）**
   - 定义：系统无故障运行的能力
   - 实现策略：冗余设计、容错机制、错误检测、检查点/回滚

3. **可用性（Availability）**
   - 定义：系统正常提供服务的时间比例
   - 实现策略：集群、故障转移、心跳机制

4. **安全性（Security）**
   - 定义：保护系统免受恶意攻击
   - 实现策略：身份认证、访问控制、加密、审计

5. **可修改性（Modifiability）**
   - 定义：系统易于修改和维护
   - 实现策略：分层、模块化、接口抽象、设计模式

6. **可扩展性（Scalability）**
   - 定义：系统处理增长的能力
   - 实现策略：水平扩展、垂直扩展、微服务

#### 2.1.3 架构评估方法

| 方法 | 全称 | 用途 | 特点 |
|------|------|------|------|
| **SAAM** | 软件架构分析方法 | 评估可修改性 | 基于场景、定性分析 |
| **ATAM** | 架构权衡分析方法 | 多质量属性评估 | 风险点、敏感点、权衡点 |
| **ALMA** | 架构级别修改分析方法 | 评估修改影响 | SAAM 的扩展 |

**ATAM 评估步骤（记忆）：**
1. 介绍 ATAM 方法
2. 介绍业务驱动因素
3. 介绍系统架构
4. 确定架构方法
5. 生成质量属性效用树
6. 分析架构方法
7. 头脑风暴和优先级排序
8. 分析架构方法（再次）
9. 呈现结果

### 专题二：系统开发（高频）

#### 2.2.1 开发方法

| 开发方法 | 特点 | 适用场景 |
|---------|------|---------|
| **结构化方法** | 自顶向下、逐步求精 | 需求明确系统 |
| **面向对象方法** | 封装、继承、多态 | 复杂业务系统 |
| **敏捷开发** | 迭代、增量、用户参与 | 需求变化快系统 |
| **DevOps** | 开发运维一体化 | 持续交付系统 |

#### 2.2.2 需求工程

**需求获取技术：**
- 访谈、问卷调查、头脑风暴
- 原型法、联合应用开发（JAD）
- 观察法、文档分析

**需求分类：**
- 业务需求 → 用户需求 → 功能需求 → 非功能需求

**非功能性需求（质量属性需求）：**
- 性能需求、可靠性需求、安全性需求、易用性需求

#### 2.2.3 建模方法

**UML 图分类（重点）：**

| 图类型 | 用途 | 包含图 |
|--------|------|--------|
| **用例图** | 功能需求建模 | 参与者、用例、关系 |
| **类图** | 静态结构建模 | 类、接口、关系 |
| **序列图** | 交互行为建模 | 对象、消息、生命线 |
| **状态图** | 对象状态变化 | 状态、转换、事件 |
| **活动图** | 业务流程建模 | 活动、决策、并发 |
| **组件图** | 物理结构建模 | 组件、接口、依赖 |
| **部署图** | 硬件部署建模 | 节点、连接 |

### 专题三：数据库与大数据（高频）

#### 2.3.1 数据库设计

**E-R 模型三要素：**
- 实体（矩形）、属性（椭圆）、关系（菱形）

**关系规范化：**
- 1NF：属性不可再分
- 2NF：消除非主属性对码的部分依赖
- 3NF：消除非主属性对码的传递依赖
- BCNF：消除主属性对码的部分和传递依赖

#### 2.3.2 NoSQL 数据库

| 类型 | 代表产品 | 适用场景 |
|------|---------|---------|
| **键值存储** | Redis、Memcached | 缓存、会话管理 |
| **文档存储** | MongoDB、CouchDB | 内容管理、日志 |
| **列族存储** | HBase、Cassandra | 大数据存储 |
| **图数据库** | Neo4j、JanusGraph | 社交网络、知识图谱 |

**Redis 常见考点：**
- 数据类型：String、List、Set、ZSet、Hash
- 持久化：RDB 快照、AOF 日志
- 应用场景：缓存、分布式锁、计数器、排行榜

#### 2.3.3 大数据架构

**Lambda 架构：**
```
数据输入 → 批处理层（Batch Layer）+ 速度层（Speed Layer） → 服务层 → 输出
```
- 批处理层：Hadoop、Spark（处理全量数据）
- 速度层：Storm、Flink（处理实时数据）
- 服务层：HBase、Cassandra（提供查询）

**Kappa 架构：**
```
数据输入 → 流处理层（Kafka + Flink） → 输出
```
- 纯流处理架构，简化维护

**大数据存储对比（2024 年考点）：**
- HDFS：适合批量读写，不适合随机读写
- HBase：适合随机读写，基于列族存储
- Hive：数据仓库，适合离线分析

### 专题四：Web 应用与云原生（高频）

#### 2.4.1 Web 架构演进

| 阶段 | 架构 | 特点 |
|------|------|------|
| 第一阶段 | 单体架构 | 全部功能打包部署 |
| 第二阶段 | 垂直拆分 | 按业务拆分独立部署 |
| 第三阶段 | SOA 架构 | 服务总线、服务复用 |
| 第四阶段 | 微服务架构 | 服务独立、去中心化 |

#### 2.4.2 微服务架构

**核心特征：**
- 服务组件化、按业务组织服务
- 产品思维而非项目思维
- 智能端点、哑管道
- 去中心化治理、去中心化数据管理
- 基础设施自动化、容错设计

**微服务关键技术栈：**
- 服务注册发现：Eureka、Nacos、Consul
- 配置中心：Config、Nacos、Apollo
- API 网关：Zuul、Gateway、Kong
- 负载均衡：Ribbon、Nginx
- 熔断器：Hystrix、Sentinel、Resilience4j
- 链路追踪：Zipkin、Sleuth、SkyWalking

#### 2.4.3 云原生架构

**云原生四大核心技术：**
1. **容器化**：Docker、Podman
2. **容器编排**：Kubernetes（K8s）
3. **微服务**：Spring Cloud、Service Mesh
4. **DevOps/CI/CD**：Jenkins、GitLab CI、ArgoCD

**云原生架构原则：**
- 服务化原则、弹性原则、可观测性原则
- 韧性原则、自动化原则、零信任安全

### 专题五：嵌入式系统（中频）

#### 2.5.1 嵌入式系统特点

- 专用性、实时性、资源受限
- 可靠性要求高、功耗敏感

#### 2.5.2 实时系统调度

| 调度算法 | 特点 | 适用场景 |
|---------|------|---------|
| **RMS** | 速率单调调度 | 周期任务、静态优先级 |
| **EDF** | 最早截止期限优先 | 动态优先级、最优算法 |
| **时间片轮转** | 公平调度 | 分时系统 |

#### 2.5.3 嵌入式软件架构

- 前后台系统（查询 + 中断）
- 轮询系统
- 实时操作系统（RTOS）

---

### 专题六：信息系统集成（高频）

#### 2.6.1 企业应用集成（EAI）

**集成层次：**
1. **数据级集成**：数据同步、ETL
2. **接口级集成**：API、Web Service
3. **业务流程级集成**：BPM、工作流
4. **界面级集成**：门户、单点登录

#### 2.6.2 SOA 架构

**SOA 三角色：**
- 服务提供者、服务消费者、服务注册中心

**SOA 关键技术：**
- Web Service（SOAP、WSDL、UDDI）
- 服务总线（ESB）
- BPEL（业务流程执行语言）

#### 2.6.3 中台架构

**中台分类：**
- 业务中台：用户中心、订单中心、支付中心
- 数据中台：数据仓库、数据湖、数据分析
- 技术中台：微服务框架、 DevOps 平台

## 三、案例分析答题技巧

### 3.1 答题步骤

**第一步：审题（5 分钟）**
- 快速浏览 4 道题，选择熟悉的 3 题
- 仔细阅读题干，标记关键信息

**第二步：分析（5 分钟/题）**
- 确定考查的知识点
- 在草稿纸上列出答题要点

**第三步：作答（35 分钟/题）**
- 条理清晰，分点作答
- 使用专业术语

### 3.2 答题模板

#### 模板一：架构风格选择题
```
根据题干描述，系统具有以下特点：
1. ...
2. ...
因此应采用 XX 架构风格。

该架构风格的优势：
1. ...
2. ...
3. ...
```

#### 模板二：质量属性分析题
```
(1) 质量属性识别：
- "系统响应时间不超过 2 秒" → 性能
- "系统可用性达到 99.9%" → 可用性
- "系统能够抵御 XX 攻击" → 安全性

(2) 实现策略：
- 性能：引入缓存、异步处理、负载均衡
- 可用性：集群部署、故障转移
- 安全性：身份认证、访问控制、数据加密
```

#### 模板三：技术对比题
```
XX 技术与 YY 技术的对比：

相同点：
1. ...

不同点：
1. XX 技术...，YY 技术...
2. XX 技术...，YY 技术...

适用场景：
- XX 技术适用于...
- YY 技术适用于...
```

### 3.3 常见答题术语

| 场景 | 术语/短语 |
|------|----------|
| 性能优化 | 引入缓存、异步处理、连接池、数据分片 |
| 可靠性提升 | 冗余设计、故障转移、心跳检测、检查点 |
| 可扩展性 | 水平扩展、微服务拆分、插件化 |
| 安全性 | 身份认证、访问控制、数据加密、审计日志 |
| 解耦 | 接口抽象、消息队列、事件驱动 |
