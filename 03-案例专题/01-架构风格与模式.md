# 专题一：架构风格与模式

> 来源：系统架构设计师知识大纲 + Gemini 对话补完 + 历年真题考点

---

## 一、核心知识体系

### 1.1 传统架构风格

| 风格 | 特点 | 适用场景 |
|------|------|---------|
| **分层架构** | 层次分明、职责分离，每层只与相邻层交互 | 企业应用系统（三层/五层架构） |
| **C/S（Client/Server，客户端/服务器）架构** | 响应快、交互性强、客户端需安装 | 桌面应用、实时系统、图形界面 |
| **B/S（Browser/Server，浏览器/服务器）架构** | 零客户端、易维护、基于 HTTP | Web 应用、互联网应用 |
| **管道-过滤器** | 数据像水流一样逐个处理单元传递，不需要等待前一步完全做完 | 编译器、实时数据处理、Unix 管道 |
| **批处理序列** | 每一步必须等前一步完全做完，数据作为一个整体传给下一阶段 | 离线日志分析、批量报表生成 |
| **主程序-子程序** | 自顶向下、逐步求精，主程序调用子程序 | 传统结构化开发 |
| **仓库风格** | 以数据为中心，组件通过中央数据仓库交互 | 数据库系统、黑板系统 |
| **微内核架构（Microkernel Architecture）** | 核心系统 + 插件扩展，核心只负责协调 | IDE、可扩展系统、插件化平台 |

**管道-过滤器 vs 批处理序列的区别（常考）：**
- 管道-过滤器：增量传递，不需要等待，支持并发
- 批处理序列：整体传递，必须等待，顺序执行

### 1.2 分布式架构风格

| 风格 | 特点 | 适用场景 |
|------|------|---------|
| **SOA（Service-Oriented Architecture，面向服务架构）** | 服务复用、ESB（Enterprise Service Bus，企业服务总线）总线、粗粒度 | 企业集成、异构系统互联 |
| **微服务架构** | 独立部署、服务自治、细粒度、去中心化数据管理 | 大型复杂系统、互联网应用 |
| **事件驱动架构（Event-Driven Architecture，EDA）** | 松耦合、异步处理、发布-订阅模式 | 消息系统、实时通知、物联网 |

**微服务核心组件（必背）：**
- **服务发现**：Eureka、Consul、Nacos —— 解决"谁在哪里"的问题
- **API（Application Programming Interface，应用程序编程接口）网关**：Zuul、Spring Cloud Gateway —— 路由、鉴权、限流
- **配置中心**：Apollo、Nacos —— 环境配置动态切换
- **断路器**：Hystrix、Sentinel —— 防止雪崩效应

**拆分原则：** 单一职责、关注点分离、围绕业务建模

### 1.3 架构描述：4+1 视图模型

架构师必须通过不同的视图向不同的人交差。

| 视图 | 关注点 | 核心对象 | 受众 | 记忆口诀 |
|------|--------|---------|------|---------|
| **逻辑视图** | 功能需求，"系统能干什么" | 类、对象、组件 | 开发人员、设计人员 | "功能" |
| **开发视图** | 软件模块组织，"代码怎么放" | 程序库、子系统、集成环境 | 程序员 | "代码" |
| **过程视图** | 并发、同步、性能 | 进程、线程 | 系统集成人员 | "运行" |
| **物理视图** | 软件到硬件的映射，"部署在哪里" | 节点、网络拓扑、服务器 | 运维人员、系统工程师 | "部署" |
| **场景视图** | 用例串联前面四个视图 | 用例、场景 | 所有利益相关者 | "灵魂" |

**易错点：** 展示系统部署在多个机房（如新加坡、法兰克福、上海）属于**物理视图**，不是场景视图。场景视图关注的是业务流程的串联（如用户从下单到支付的流程）。

### 1.4 架构评估方法

| 方法 | 全称 | 用途 | 特点 |
|------|------|------|------|
| **SAAM（Software Architecture Analysis Method，软件架构分析方法）** | 评估可修改性 | 基于场景、定性分析 |
| **ATAM（Architecture Tradeoff Analysis Method，架构权衡分析方法）** | 多质量属性评估 | 风险点、敏感点、权衡点、非风险点 |
| **ALMA（Architecture-Level Modifiability Analysis，架构级别修改分析方法）** | 评估修改影响 | SAAM 的扩展 |

**ATAM 评估 9 步骤（记忆口诀：业介系质头脑现）：**
1. 介绍 ATAM 方法
2. 介绍业务驱动因素
3. 介绍系统架构
4. 确定架构方法
5. 生成质量属性效用树
6. 分析架构方法
7. 头脑风暴和优先级排序
8. 分析架构方法（再次）
9. 呈现结果

**ATAM 四个关键概念：**
- **风险点 (Risk)**：可能导致质量属性无法实现的架构决策
- **非风险点 (Non-Risk)**：经过验证不会造成问题的决策
- **敏感点 (Sensitivity Point)**：对某个质量属性影响显著的参数
- **权衡点 (Tradeoff Point)**：影响多个质量属性且相互矛盾的参数

### 1.6 可靠性模式

| 模式 | 原理 | 适用场景 |
|------|------|---------|
| **Ping/Echo（心跳检测）** | 主动发送 Ping 请求，等待 Echo 响应，超时判定故障 | 分布式节点存活检测、心跳监控 |
| **心跳模式（Heartbeat）** | 定期发送心跳信号，接收方未收到心跳则判定故障 | 主从切换、集群成员管理 |
| **冗余设计** | 关键组件配置多个副本，一个故障自动切换 | 高可用系统（并联系统） |

**考点注意**：2024 年下考了 ping/echo 与心跳模式在可靠性场景中的应用。

### 1.5 架构描述语言（ADL，Architecture Description Language）

- **定义**：形式化语言，用来描述软件架构（组件、连接件、配置）
- **三要素**：
  - **组件 (Components)**：计算或数据的单元
  - **连接件 (Connectors)**：组件间的交互关系
  - **配置 (Configurations)**：组件和连接件组成的拓扑结构
- 偶尔出现在选择题中，属于冷门考点

## 三、六大质量属性（核心重点）

| 质量属性 | 定义 | 实现策略 |
|---------|------|---------|
| **性能** | 系统响应时间和吞吐量 | 缓存、异步处理、负载均衡、数据分片 |
| **可靠性** | 系统无故障运行的能力 | 冗余设计、容错机制、错误检测、检查点/回滚 |
| **可用性** | 系统正常提供服务的时间比例 | 集群、故障转移、心跳机制 |
| **安全性** | 保护系统免受恶意攻击 | 身份认证、访问控制、加密、审计 |
| **可修改性** | 系统易于修改和维护 | 分层、模块化、接口抽象、设计模式 |
| **可扩展性** | 系统处理增长的能力 | 水平扩展、垂直扩展、微服务 |

**质量属性场景六组成部分：** 刺激源、刺激、制品、环境、响应、响应度量

## 四、答题模板

### 架构风格比较题答题套路

```
答题框架：
1. 分别描述两种/多种架构风格的核心特点
2. 从以下维度进行对比：
   - 数据处理方式
   - 可扩展性
   - 性能
   - 可修改性
   - 灵活性/维护性
3. 结合案例场景，给出选型建议及理由
```

### 质量属性识别题答题套路

```
1. 从案例描述中提取关键词（如"响应时间"→性能，"不停机"→可用性）
2. 对应六大质量属性
3. 给出实现策略（使用上面的表格）
4. 构建效用树：场景→属性→优先级
```
