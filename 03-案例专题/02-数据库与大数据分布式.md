## 一、核心知识体系

### 1.1 数据库设计基础

**E-R 模型三要素：**
- **实体（Entity）**：矩形，表示现实世界中可区分的事物（如学生、课程）
- **属性（Attribute）**：椭圆，描述实体的特征（如学号、姓名）
- **关系（Relationship）**：菱形，表示实体之间的联系（如选课、授课）
- **三类关系**：一对一（1:1）、一对多（1:N）、多对多（M:N）

**关系规范化（范式理论）：**

| 范式 | 定义 | 解决的问题 |
|------|------|-----------|
| **1NF** | 属性不可再分（原子性） | 消除重复组 |
| **2NF** | 在 1NF 基础上，消除非主属性对码的部分依赖 | 消除部分依赖 |
| **3NF** | 在 2NF 基础上，消除非主属性对码的传递依赖 | 消除传递依赖 |
| **BCNF** | 在 3NF 基础上，消除主属性对码的部分和传递依赖 | 更严格的约束 |

**反规范化设计：**
- 在大数据场景下，为了提高查询性能，**故意违反范式**，增加冗余数据
- 常见手段：增加冗余列、派生列、宽表设计
- 权衡：用空间换时间，写入性能下降、查询性能提升

### 1.2 OLTP（Online Transaction Processing，联机事务处理）vs OLAP（Online Analytical Processing，联机分析处理）vs HTAP（Hybrid Transactional/Analytical Processing，混合事务/分析处理）

| 类型 | 全称 | 目标 | 关注点 | 代表技术 |
|------|------|------|--------|---------|
| **OLTP** | 联机事务处理 | 快、准、稳 | ACID（Atomicity 原子性、Consistency 一致性、Isolation 隔离性、Durability 持久性）事务性 | MySQL、PostgreSQL |
| **OLAP** | 联机分析处理 | 大、深、全 | 查询吞吐量 | ClickHouse、Doris、Hive |
| **HTAP** | 混合事务/分析处理 | 一套系统同时搞定 | 先进架构（加分项） | TiDB |

**架构师的深度思考**：数据从 OLTP 搬到 OLAP 的过程称为 **ETL（Extract, Transform, Load，提取-转换-加载）**。当前流行趋势是用 Canal 监听 Binlog 实现 **CDC（Change Data Capture，变更数据捕获）**。

### 1.2 传统 RDBMS（Relational Database Management System，关系型数据库管理系统）性能优化（三层递进）

#### 第一层：基础优化
- **索引优化**：B+ 树、覆盖索引、联合索引
- **查询优化**：避免全表扫描、减少 Join

#### 第二层：读写分离
- **原理**：主库（Master）负责写，从库（Slave）负责读，通过主从复制同步
- **解决**：数据库负载均衡，缓解读压力
- **考点**：主从延迟问题、读写一致性问题

#### 第三层：分库分表（Sharding）
- **垂直切分**：按业务拆分（用户表、订单表分不同库）
- **水平切分**：按记录拆分（订单表按 ID 取模分散到多个表）
- **三大挑战**：
  1. **分片键选择**：影响查询效率和数据倾斜
  2. **全局唯一 ID**：雪花算法 (Snowflake) 或 UUID（Universally Unique Identifier，通用唯一识别码）
  3. **分布式 Join**：跨节点关联性能极差 → 用**宽表设计（反范式）** 替代

### 1.3 缓存架构（Redis）

**五种数据类型：**
- **String**：计数器、Session（会话）存储
- **List**：消息队列、排行榜
- **Set**：标签、共同好友
- **ZSet**：排行榜（带分数排序）
- **Hash**：对象存储（用户信息）

**持久化：**
- **RDB（Redis Database Backup，快照持久化）**：快照，恢复快，可能丢数据
- **AOF（Append Only File，追加日志持久化）**：日志，数据更安全，恢复慢

**三大缓存问题（必考）：**

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 查询不存在的数据，缓存和 DB 都不命中 | 布隆过滤器、缓存空值 |
| **缓存击穿** | 热点 Key 过期，大量请求打到 DB | 互斥锁、逻辑过期 |
| **缓存雪崩** | 大量 Key 同时过期或 Redis 宕机 | 随机过期时间、集群高可用 |

**缓存一致性方案：**
- **Cache-Aside**（旁路缓存）：先读缓存，不命中读 DB 写缓存；更新时先更新 DB 再删除缓存
- **延迟双删**：先删缓存 → 更新 DB → 延迟后再次删除缓存
- **订阅 Binlog**：通过 Canal 监听数据库变更，异步更新缓存

**缓存分片（一致性哈希）：**
- **问题**：单台 Redis 容量有限，需要分片存储
- **一致性哈希环**：将哈希值映射为 0~2³²⁻¹ 的环，数据和服务节点都在环上
- **虚拟节点**：解决数据倾斜问题，每个物理节点映射多个虚拟节点
- **优点**：节点增减时只影响相邻节点，不需要重新哈希全部数据

**布隆过滤器（Bloom Filter）：**
- **原理**：位数组 + 多个哈希函数，将元素映射到位数组的多个位置
- **特性**："不存在一定不存在，存在可能存在"（有假阳性，无假阴性）
- **场景**：缓存穿透防护、垃圾邮件过滤、URL 去重
- **优点**：空间效率极高，查询时间 O(1)；**缺点**：不支持删除

**Redis 主从复制：**
- **全量同步**：主库执行 BGSAVE 生成 RDB 快照，传送给从库，从库加载
- **增量同步**：复制积压缓冲区（repl-backlog-buffer）记录写偏移量（repl-offset），网络恢复后补传
- **断线重连**：从库发送 PSYNC 命令，根据偏移量决定全量还是增量

**Redis 内存淘汰机制（8 种策略）：**
- **noeviction**：不淘汰，写操作报错（默认）
- **allkeys-lru**：所有 key 中使用 LRU（Least Recently Used，最近最少使用）淘汰
- **volatile-lru**：仅淘汰设置了过期时间的 key
- **allkeys-lfu**：所有 key 中使用 LFU（Least Frequently Used，最不经常使用）淘汰
- **volatile-lfu**：仅淘汰设置了过期时间的 key，按 LFU
- **allkeys-random**：随机淘汰
- **volatile-random**：仅淘汰设置了过期时间的 key，随机淘汰
- **volatile-ttl**：淘汰 TTL（Time To Live，存活时间）最短的 key

### 1.4 NoSQL（Not Only SQL，非关系型数据库）数据库选型

| 类型 | 代表产品 | 应用场景 | 优点 |
|------|---------|---------|------|
| **键值 (KV)** | Redis、Memcached | Session、排行榜、计数器、热点缓存 | 极致性能、原子操作 |
| **文档型** | MongoDB、CouchDB | 用户动态、商品详情（属性不固定）、日志存储 | Schema-less、水平扩展 |
| **列族型** | HBase、Cassandra | 海量历史轨迹、监控数据 | 写吞吐量极大、离线分析友好 |
| **图数据库** | Neo4j、JanusGraph | 社交关系、风险审计、推荐引擎 | 处理多表深度关联效率极高 |

**答题套路**：考题常问"为什么选这个 NoSQL 而不是关系型？"—— 根据业务场景对应类型 + 说明优势。

### 1.5 大数据处理架构

#### Lambda vs Kappa vs 湖仓一体

| 架构 | 核心思想 | 适用场景 | 缺点 |
|------|---------|---------|------|
| **Lambda** | 批处理层（保准）+ 实时流层（保快）→ 服务层合并 | 既要准确又要实时的场景 | 维护两套代码 |
| **Kappa** | 纯流式处理，全量数据重放 | 纯实时场景 | 不适合批处理 |
| **湖仓一体 (Lakehouse)** | 在数据湖的存储上实现数据仓库的管理 | 实时入湖 + 并发查询 | 技术栈较新 |

**湖仓一体代表技术**：Iceberg、Hudi（支持 ACID、支持海量数据实时入湖和并发查询）

**数据流链路**：采集 (CDC/Binlog) → 消息队列 (Kafka) → 计算 (Flink) → 存储 (Iceberg/HDFS/S3)

### 1.6 分布式事务（架构师分水岭）

| 方案 | 适用场景 | 核心思想 | 优缺点 |
|------|---------|---------|--------|
| **2PC/3PC** | 强一致性场景 | 协调者统一调度，所有参与者投票 | 阻塞、性能差 |
| **TCC** | 金融级、高一致性 | Try(预留)→Confirm(执行)→Cancel(回滚) | 侵入性强、需处理幂等性/悬挂 |
| **Saga** | 长事务、多参与者 | 拆分为小事务，失败时执行补偿操作 | 不保证隔离性、可能看到中间状态 |
| **可靠消息最终一致性** | 高并发互联网场景 | 本地事务 + MQ 半消息 → 下游消费 | 最终一致、延迟可接受、最常用 |

**可靠消息最终一致性流程（配合 RocketMQ/Kafka）：**
1. 服务 A 执行本地事务并发送"半消息"到 MQ
2. 本地事务成功后提交消息
3. 服务 B 监听消息执行本地事务，失败则重试

**TCC 三阶段深入：**
- **Try**：资源检测与预留（冻结资金，不直接扣款）
- **Confirm**：使用预留资源执行真正操作
- **Cancel**：任何 Try 失败时释放预留资源

### 1.7 一致性协议与理论

| 理论/协议 | 核心内容 | 考点 |
|-----------|---------|------|
| **CAP（Consistency 一致性、Availability 可用性、Partition tolerance 分区容忍性）定理** | 三者最多选二 | CP 场景选 Raft，AP 场景选最终一致性 |
| **BASE（Basically Available 基本可用、Soft state 软状态、Eventually consistent 最终一致性）理论** | 牺牲强一致性换取可用性 | 社交/电商选 BASE |
| **PACELC（Partition And Consistency Else Latency And Consistency）定理** | CAP 的扩展：无分区时在延迟和一致性之间权衡 | 全球访问场景的延迟优化 |
| **Paxos/Raft** | 分布式共识协议，超过半数节点同意即写入成功 | 考"共识"选 Raft |

### 1.8 数据迁移与冷热分离

**零停机迁移四阶段：**
1. **双写阶段**：新旧库同时写入，以旧库为准
2. **历史数据搬迁**：CDC (Canal/Debezium) 同步存量数据
3. **数据校验**：比对新旧库差异并修补
4. **切换读写**：先切读，后停旧库写

**存储分级 (Tiered Storage)：**
- **热数据**（7天~1个月）：SSD/内存，高并发
- **温数据**（1~6个月）：HDD，偶尔查询
- **冷数据**（半年以上）：对象存储 (S3/OSS) + Parquet/ORC 列式存储

**CQRS（Command Query Responsibility Segregation，命令查询职责分离）：**
- 核心库只负责 INSERT/UPDATE（少索引，保 TPS）
- 复杂查询交给 Elasticsearch（通过 Binlog/MQ（Message Queue，消息队列）异步同步）

## 三、案例分析答题模板

### 数据库性能瓶颈演进方案（万能答题路径）

```
短期方案：引入 Redis 缓存，缓解读压力；优化索引
中期方案：实施读写分离（一主多从）；垂直分库，按业务解耦
长期方案：水平分表（Sharding）；引入分布式数据库（如 TiDB）或 NoSQL
         （根据数据类型选型）；建立大数据平台（湖仓一体）进行冷热分离
```

### NoSQL 选型答题模板

```
根据业务特点选择 XX 数据库：
- 业务特征：数据量大/属性不固定/高并发写入/深度关联查询
- 对应 NoSQL 类型：KV/文档/列族/图
- 选型理由：XX 类型具备 XX 优势，相比关系型数据库在 XX 场景下性能提升显著
```

### 分布式事务答题模板

```
根据业务场景选择 XX 方案：
- 金融级高一致：TCC（需处理幂等性、悬挂问题）
- 长事务多环节：Saga（补偿机制）
- 高并发互联网：可靠消息最终一致性（MQ半消息+重试）
- 核心理论依据：BASE 理论，牺牲强一致性换取可用性
```
