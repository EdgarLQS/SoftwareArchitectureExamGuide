# Redis 专题复习指南

## 一、Redis 概述

### 1.1 Redis 简介

**Redis**（Remote Dictionary Server）是一个开源的、基于内存的键值存储系统，常用作数据库、缓存和消息中间件。

**核心特点：**
- **基于内存**：数据存储在内存中，读写速度极快（10 万 + QPS）
- **键值存储**：Key-Value 数据结构，Key 必须是字符串
- **数据类型丰富**：支持 String、List、Set、ZSet、Hash 等
- **持久化支持**：RDB 快照 + AOF 日志
- **高可用**：支持主从复制、哨兵模式、集群
- **单线程模型**：基于 IO 多路复用，避免上下文切换

### 1.2 Redis 应用场景

| 场景 | 数据类型 | 说明 |
|------|---------|------|
| **缓存** | 所有类型 | 热点数据缓存、会话管理 |
| **分布式锁** | String | SETNX 实现互斥锁 |
| **计数器** | String | 点赞数、访问量、秒杀库存 |
| **排行榜** | ZSet | 游戏排行榜、热搜榜 |
| **消息队列** | List/Stream | 异步任务、延迟队列 |
| **购物车** | Hash | 用户购物车数据 |
| **社交关系** | Set | 共同好友、关注列表 |
| **地理位置** | Geo | 附近的人、位置追踪 |

## 二、Redis 数据结构（必考）

### 2.1 五大基本数据类型

#### 2.1.1 String（字符串）

**特点：**
- 二进制安全，最大存储 512MB
- 支持自增自减操作

**常用命令：**
```bash
SET key value              # 设置值
GET key                    # 获取值
SETNX key value            # 分布式锁（不存在才设置）
SETEX key seconds value    # 设置值 + 过期时间
INCR key                   # 自增 1
INCRBY key increment       # 自增指定值
DECR key                   # 自减 1
MSET key1 value1 key2 value2  # 批量设置
MGET key1 key2             # 批量获取
```

**应用场景：**
- 缓存用户信息、商品信息
- 分布式锁（SETNX）
- 计数器（点赞、浏览、秒杀）

---

#### 2.1.2 List（列表）

**特点：**
- 双向链表实现
- 元素有序、可重复
- 支持两端推送和弹出

**常用命令：**
```bash
LPUSH key value1 value2    # 左侧插入
RPUSH key value1 value2    # 右侧插入
LPOP key                   # 左侧弹出
RPOP key                   # 右侧弹出
LRANGE key start end       # 获取范围元素
LLEN key                   # 获取列表长度
LINDEX key index           # 获取指定索引元素
```

**应用场景：**
- 消息队列（LPUSH + RPOP）
- 最新列表（最新文章、最新动态）
- 栈和队列实现

---

#### 2.1.3 Hash（哈希）

**特点：**
- 键值对集合，适合存储对象
- 字段可独立修改

**常用命令：**
```bash
HSET key field value       # 设置字段值
HGET key field             # 获取字段值
HMSET key field1 v1 field2 v2  # 批量设置
HMGET key field1 field2    # 批量获取
HGETALL key                # 获取所有字段
HDEL key field             # 删除字段
HLEN key                   # 获取字段数量
HINCRBY key field increment # 字段自增
```

**应用场景：**
- 购物车（用户 ID 为 key，商品 ID 为 field）
- 用户画像（用户 ID 为 key，标签为 field）
- 对象存储（比 String 更灵活）

---

#### 2.1.4 Set（集合）

**特点：**
- 无序集合，元素唯一
- 支持交集、并集、差集运算

**常用命令：**
```bash
SADD key member1 member2   # 添加元素
SREM key member            # 删除元素
SMEMBERS key               # 获取所有元素
SISMEMBER key member       # 判断元素是否存在
SCARD key                  # 获取元素数量
SINTER key1 key2           # 交集
SUNION key1 key2           # 并集
SDIFF key1 key2            # 差集
```

**应用场景：**
- 好友关系、关注列表
- 共同好友（交集）
- 可能认识的人（差集）
- 抽奖系统（随机抽取）

---

#### 2.1.5 ZSet（有序集合）

**特点：**
- 有序集合，元素唯一
- 每个元素关联一个分数（score）
- 按分数排序

**常用命令：**
```bash
ZADD key score member1 score2 member2  # 添加元素
ZREM key member            # 删除元素
ZRANGE key start end [WITHSCORES]  # 按分数升序获取
ZREVRANGE key start end    # 按分数降序获取
ZRANK key member           # 获取排名
ZCARD key                  # 获取元素数量
ZINCRBY key increment member # 分数自增
ZCOUNT key min max         # 统计分数范围内的元素
```

**应用场景：**
- 排行榜（游戏、销售、热搜）
- 优先级队列
- 延迟队列（时间戳为 score）

---

### 2.2 三大特殊数据类型

#### 2.2.1 HyperLogLog

**特点：**
- 基数统计（去重计数）
- 占用空间固定（12KB）
- 有误差（0.81%）

**常用命令：**
```bash
PFADD key element1 element2  # 添加元素
PFCOUNT key                  # 获取基数
PFMERGE destkey sourcekey    # 合并
```

**应用场景：**
- UV 统计（网站独立访客）
- 海量数据去重

---

#### 2.2.2 Bitmap（位图）

**特点：**
- 按位存储，极致节省空间
- 适合布尔值场景

**常用命令：**
```bash
SETBIT key offset value    # 设置位值
GETBIT key offset          # 获取位值
BITCOUNT key               # 统计 1 的数量
BITOP op destkey key1 key2 # 位运算
```

**应用场景：**
- 用户签到
- 状态标记（在线/离线）
- 布隆过滤器基础

---

#### 2.2.3 Geo（地理位置）

**特点：**
- 存储经纬度坐标
- 支持距离计算、附近搜索

**常用命令：**
```bash
GEOADD key longitude latitude member  # 添加位置
GEODIST key member1 member2           # 计算距离
GEORADIUS key x y radius              # 指定半径搜索
GEORADIUSBYMEMBER key member radius   # 某成员附近搜索
```

**应用场景：**
- 附近的人
- 网约车位置
- 物流追踪

---

### 2.2.4 布隆过滤器（Bloom Filter）

**特点：**
- 基于 Bitmap 实现的概率型数据结构
- 位数组 + k 个独立哈希函数
- 核心特性：**"不存在一定不存在，存在可能存在"**
- 有假阳性（False Positive），无假阴性（False Negative）

**常用命令：**
```bash
BF.ADD key element             # 添加元素
BF.EXISTS key element          # 判断元素是否存在
BF.MADD key element1 element2  # 批量添加
BF.MEXISTS key element1 element2  # 批量判断
```

**优缺点对比：**

| 优点 | 缺点 |
|------|------|
| 空间效率极高（亿级数据仅需几 MB） | 存在假阳性（误判） |
| 查询时间 O(1)，极快 | 不支持删除（需 Counting Bloom 变体） |
| 无假阴性（漏判率为 0） | 哈希函数质量影响准确率 |

**应用场景：**
- 缓存穿透防护（查询前预判断）
- 垃圾邮件过滤
- URL 去重
- 海量数据黑名单

**2022年真题考点：** 黑客故意发送不存在的商品配送时间查询导致缓存穿透，使用布隆过滤器拦截非法请求。

## 三、Redis 持久化（必考）

### 3.1 RDB（Redis Database）

**原理：**
- 定时将内存数据快照保存到磁盘
- 文件：dump.rdb
- 触发方式：手动触发（SAVE/BGSAVE）或自动触发（配置）

**配置示例：**
```conf
save 900 1        # 900 秒内至少 1 个 key 变化
save 300 10       # 300 秒内至少 10 个 key 变化
save 60 10000     # 60 秒内至少 10000 个 key 变化
```

**优点：**
- 恢复速度快
- 文件紧凑，适合备份
- 不影响主进程（BGSAVE）

**缺点：**
- 可能丢失最后一次快照后的数据
- 大数据量时 fork 子进程耗时

---

### 3.2 AOF（Append Only File）

**原理：**
- 记录每次写操作命令
- 重启时重放命令恢复数据
- 支持三种同步策略

**同步策略：**
| 策略 | 配置 | 数据安全性 | 性能 |
|------|------|-----------|------|
| 每秒同步 | everysec | 丢失 1 秒数据 | 推荐 |
| 每次同步 | always | 不丢失 | 差 |
| 从不同步 | no | 丢失最多 | 最好 |

**配置示例：**
```conf
appendonly yes                    # 开启 AOF
appendfsync everysec              # 每秒同步
auto-aof-rewrite-percentage 100   # AOF 文件增长 100% 时重写
```

**优点：**
- 数据更安全（最多丢失 1 秒）
- 可读性强（命令文本）

**缺点：**
- 文件体积大
- 恢复速度慢于 RDB

---

### 3.3 混合持久化（Redis 4.0+）

**原理：**
- RDB + AOF 结合
- AOF 重写时，将当前内存以 RDB 格式写入 AOF 文件头部
- 后续增量数据以 AOF 格式追加

**优点：**
- 兼具 RDB 恢复快和 AOF 数据安全
- 重写效率高

---

### 3.4 持久化选型建议

| 场景 | 推荐方案 |
|------|---------|
| 缓存（可丢失） | 关闭持久化 |
| 一般业务 | RDB + AOF（everysec） |
| 高安全要求 | AOF（always） |
| 大数据量 | 混合持久化 |

## 四、Redis 高可用架构（必考）

### 4.1 主从复制

**特点：**
- 一主多从架构
- 主库写，从库读（读写分离）
- 从库故障可切换

**配置：**
```conf
# 从库配置
replicaof 127.0.0.1 6379    # 指定主库
```

**复制过程：**

**全量同步（首次连接或断线时间过长）：**
```
  Slave                    Master
    |                        |
    |--- (1) PSYNC --------->|  ① 从库发送 PSYNC 命令
    |                        |-- (2) BGSAVE -------|
    |                        |  ② 主库执行 BGSAVE 生成 RDB 快照
    |                        |  ③ 同时将新写操作写入复制缓冲区（repl-backlog-buffer）
    |<- (3) RDB 文件 --------|  ④ 主库发送 RDB 快照给从库
    |-- (4) 加载 RDB -------|  ⑤ 从库加载 RDB，恢复数据
    |<- (5) 增量命令 -------|  ⑥ 主库发送缓冲区中积累的写命令
    |-- (6) 执行增量命令 ---|  ⑦ 从库执行增量命令，完成同步
    |                        |
```

**增量同步（网络短暂断开后重连）：**
- 主库维护**复制积压缓冲区（repl-backlog-buffer）**，固定大小的环形缓冲区
- 主从各自维护**复制偏移量（repl-offset）**，记录已同步的数据位置
- 断线重连时从库发送 `PSYNC <offset>`，主库判断：
  - offset 仍在缓冲区中 → **增量同步**（仅补传差异命令）
  - offset 已被覆盖 → **全量同步**（重新 BGSAVE）

```
  Client → Master: 写操作
  Master → repl-backlog-buffer: 写入命令 + 更新 offset
  Master → Slave: 通过复制流发送写命令
  Slave: 执行命令 + 更新自身 offset
```

---

### 4.2 哨兵模式（Sentinel）

**作用：**
- 监控主从节点状态
- 自动故障转移
- 配置中心（通知客户端新主库）

**架构：**
```
        Sentinel 集群（3 节点）
           /    |    \
          /     |     \
      Master -- Slave1 -- Slave2
```

**故障转移流程：**
1. 哨兵监控到主库下线
2. 哨兵之间确认（quorum）
3. 选举一个哨兵执行故障转移
4. 选择一个从库晋升为主库
5. 通知其他从库和新主库同步
6. 通知客户端

**配置：**
```conf
sentinel monitor mymaster 127.0.0.1 6379 2  # quorum=2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 60000
```

---

### 4.3 集群模式（Cluster）

**特点：**
- 去中心化，多主节点
- 数据分片存储（16384 个 slot）
- 自动故障转移

**架构：**
```
  Master1(0-5460)    Master2(5461-10922)   Master3(10923-16383)
     /    \             /    \                /    \
  Slave1 Slave2    Slave3 Slave4        Slave5 Slave6
```

**数据分片：**
- 16384 个槽（slot）
- Key 通过 CRC16 计算槽号：`CRC16(key) % 16384`
- 每个主节点负责部分槽

**客户端访问：**
- 客户端连接任意节点
- 如果 Key 不在该节点，返回 MOVED 错误并重定向
- 智能客户端会缓存槽分布

---

### 4.4 高可用方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **主从** | 简单、读写分离 | 手动切换 | 小规模 |
| **哨兵** | 自动故障转移 | 主库写瓶颈 | 中等规模 |
| **集群** | 高并发、高可用 | 复杂 | 大规模 |

---

### 4.5 缓存分片与一致性哈希

当单节点 Redis 无法满足容量和并发需求时，需要多节点集群部署。

#### 4.5.1 普通哈希分片

**原理：** `hash(key) % N`（N 为节点数）

**缺点：** 节点增减时 N 变化，导致**绝大部分 Key 需要重新映射**，缓存大面积失效（缓存重新击穿）。

#### 4.5.2 一致性哈希

**原理：**
- 将哈希值映射为 **0 ~ 2³² - 1** 的环形空间
- 数据 Key 和 Redis 节点都通过哈希函数映射到环上
- 数据沿顺时针方向找到第一个节点
- 节点增减时**只影响相邻节点**，不需要重新哈希全部数据

**一致性哈希环示意：**
```
         ┌─────────────────────┐
         │   2³² - 1 (0)       │
         │        ·            │
         │   Node A            │
         │    / \              │
         │   /   \  Key1       │
    Key4 ·     /    \          │
         |    /      · Node B  │
         |   /        /        │
         |  /  Key2  /         │
         | ·       /           │
         |  \     /    Key3    │
         |   \   /             │
         │    · Node C         │
         └─────────────────────┘
```

**虚拟节点：**
- 每个物理节点映射多个虚拟节点（如 150 个）
- 解决节点分布不均导致的数据倾斜
- 物理节点 A → 虚拟节点 A#1, A#2, A#3... 分布在环的不同位置

#### 4.5.3 分片方案对比

| 方案 | 原理 | 扩容影响 | 适用场景 |
|------|------|---------|---------|
| 普通哈希 | hash(key) % N | 全量重新分配 | 小规模固定节点 |
| 一致性哈希 | 哈希环 + 虚拟节点 | 只影响相邻节点 | Redis 集群 |
| 范围分片 | 按区间划分 | 追加新片即可 | 范围查询频繁 |
| 地理分片 | 按地域划分 | 追加新片即可 | 就近访问低延迟 |

## 五、缓存应用场景（高频考点）

### 5.1 缓存模式

#### 5.1.1 Cache-Aside（旁路缓存）

**流程：**
```
读：先读缓存 → 命中返回 → 未命中读数据库 → 写入缓存 → 返回
写：先写数据库 → 删除缓存
```

**代码示例：**
```java
// 读
String value = redis.get(key);
if (value == null) {
    value = db.query(key);
    redis.set(key, value);
}
return value;

// 写
db.update(key, value);
redis.del(key);  // 删除而非更新，避免并发问题
```

**优点：**
- 简单易懂
- 删除缓存比更新缓存更安全

**缺点：**
- 写操作后缓存未命中，影响性能

**适用：** 读多写少的场景

---

#### 5.1.2 Read/Write Through（穿透缓存）

**流程：**
```
读/写：操作缓存 → 缓存自动同步数据库
```

**特点：**
- 缓存和数据库整合为一个整体
- 由缓存层负责数据同步
- Redis 本身不支持，需要二次封装

---

#### 5.1.3 Write Behind（异步缓存写入）

**流程：**
```
写：只写缓存 → 异步批量写数据库
读：读缓存 → 未命中读数据库
```

**特点：**
- 写性能极高
- 可能丢失数据（缓存宕机）

**适用：** 高吞吐、可容忍丢失的场景

---

### 5.2 缓存三大问题（必考）

#### 5.2.1 缓存穿透

**问题描述：**
- 查询不存在的数据
- 缓存不命中，请求打到数据库
- 恶意攻击导致数据库压力大

**解决方案：**
1. **布隆过滤器**：快速判断 Key 是否存在
2. **缓存空对象**：即使不存在也缓存 null 值（设置短 TTL）
3. **接口限流**：限制同一 IP 的请求频率

**代码示例（缓存空值）：**
```java
String value = redis.get(key);
if ("NULL".equals(value)) {
    return null;  // 直接返回
}
if (value == null) {
    value = db.query(key);
    if (value == null) {
        redis.setex(key, 300, "NULL");  // 缓存空值 5 分钟
    } else {
        redis.set(key, value);
    }
}
```

---

#### 5.2.2 缓存击穿

**问题描述：**
- 热点 Key 突然过期
- 大量并发请求同时打到数据库
- 数据库瞬间压力大

**解决方案：**
1. **互斥锁**：只让一个线程查数据库，其他等待
2. **永不过期**：热点数据不设置 TTL
3. **提前预热**：过期前主动刷新
4. **逻辑过期**：返回过期数据，后台异步刷新（2025.11 新考点）

**代码示例（互斥锁）：**
```java
String value = redis.get(key);
if (value == null) {
    // 获取分布式锁
    if (redis.setnx(lockKey, "1") == 1) {
        try {
            // 双重检查
            value = redis.get(key);
            if (value == null) {
                value = db.query(key);
                redis.set(key, value);
            }
        } finally {
            redis.del(lockKey);
        }
    } else {
        // 等待后重试
        Thread.sleep(50);
        return query(key);
    }
}
```

#### 物理过期 vs 逻辑过期方案对比（2025.11 真题考点）

| 维度 | 互斥锁物理过期 | 逻辑过期 |
|------|---------------|---------|
| TTL 设置 | Redis 原生 TTL，过期自动删除 | 不设置 TTL，业务字段内嵌过期时间 |
| 数据存储 | 仅业务数据 | 业务数据 + 逻辑过期时间戳 |
| 过期判定 | Redis 自动判断 | 应用层比对时间戳 |
| 未命中行为 | 获取锁 → 查 DB → 写缓存 → 释放锁 | 获取锁 → 立即返回旧数据 → 后台异步刷新 |
| 一致性 | 强（数据不过期不返回） | 弱（可能返回过期数据） |
| 性能 | 低（争锁时阻塞等待） | 高（非阻塞，立即返回） |
| 内存 | 低（无额外字段） | 高（每个 Key 多存时间戳） |
| 适用场景 | 数据一致性要求高 | 读多写少、可接受最终一致 |

**逻辑过期查询流程：**
```
1. 查询缓存，命中数据
2. 检查逻辑过期时间：
   - 未过期 → 直接返回
   - 已过期 → 尝试获取重建锁
     - 获取成功 → 立即返回旧数据，启动后台线程异步刷新
     - 获取失败 → 返回旧数据（其他线程正在刷新）
3. 后台线程：查 DB → 更新缓存 + 新过期时间 → 释放锁
```

#### 5.2.3 缓存雪崩

**问题描述：**
- 大量 Key 同时过期
- 或缓存服务宕机
- 请求全部打到数据库

**解决方案：**
1. **随机 TTL**：过期时间增加随机值
2. **高可用架构**：哨兵/集群
3. **限流降级**：数据库层限流
4. **多级缓存**：本地缓存 + Redis

**代码示例（随机 TTL）：**
```java
// 基础时间 + 随机时间（±10%）
int ttl = 3600 + new Random().nextInt(720);
redis.setex(key, ttl, value);
```

### 5.3 缓存一致性

**问题：**
- 数据库和缓存数据不一致

**解决方案：**

| 方案 | 做法 | 一致性强度 | 性能影响 | 适用场景 |
|------|------|-----------|---------|---------|
| **过期淘汰** | 设置 TTL，过期后重新加载 | 弱（最终一致） | 最低 | 可容忍短暂不一致 |
| **主动删除** | 更新 DB 后主动删除缓存 | 中 | 低 | Cache-Aside 默认推荐 |
| **订阅 Binlog** | 监听 DB Binlog 自动更新缓存 | 强 | 中（需 Canal 等工具） | 高并发最终一致 |
| **双写同步** | 同时写 DB 和缓存 | 最强 | 高（需分布式事务） | 强一致性要求 |

**Canal + Binlog 异步更新流程：**
```
  Application → MySQL: 写操作
  MySQL → Binlog: 记录变更
  Canal → Binlog: 监听并捕获变更
  Canal → Redis: 异步更新/删除缓存
```

**推荐方案：Cache-Aside + 删除缓存**
- 写操作：先写数据库，再删除缓存
- 读操作：先读缓存，未命中读数据库并回写

### 5.4 多级缓存架构

高性能系统通常采用多级缓存架构，将请求逐层分流：

#### 5.4.1 四级缓存层次

| 层次 | 技术 | 特点 | 适用场景 |
|------|------|------|---------|
| **浏览器缓存** | HTTP Cache-Control、ETag | 离用户最近，减少网络请求 | 静态资源、API 响应 |
| **CDN 缓存** | 边缘节点缓存 | 地理就近，减轻源站压力 | 图片、视频、静态页面 |
| **应用层缓存** | Redis、Memcached、Caffeine | 进程内/外缓存，毫秒级响应 | 热点数据、会话存储 |
| **数据库缓存** | 查询缓存、缓冲池 | 数据库内置，自动管理 | 频繁查询结果 |

#### 5.4.2 分流-提速-减压-兜底框架

| 阶段 | 技术手段 | 效果 |
|------|---------|------|
| **分流** | CDN 分流静态资源、负载均衡分发请求 | 减轻源站压力 |
| **提速** | 多级缓存（浏览器→CDN→Redis→本地）、静态化页面 | 加速响应 |
| **减压** | 异步处理（MQ）、批量写入、连接池 | 降低数据库压力 |
| **兜底** | 限流降级、熔断降级、服务隔离 | 防止系统雪崩 |

#### 5.4.3 L1 + L2 + L3 组合示例

```
L1: Caffeine 本地缓存（进程内，纳秒级）
    ↓ 未命中
L2: Redis 分布式缓存（进程外，毫秒级）
    ↓ 未命中
L3: MySQL 数据库（持久化存储）
```

| 方案 | 原理 | 适用并发量 | 说明 |

### 5.6 分片策略对比

| 策略 | 原理 | 优点 | 缺点 | 扩容难度 |
|------|------|------|------|---------|
| **Hash 分片** | `hash(key) % N` | 数据分布均匀 | 扩容需全量重分配 | 高 |
| **一致性 Hash** | 哈希环 + 虚拟节点 | 扩容只影响相邻节点 | 实现复杂 | 低 |
| **范围分片** | 按区间划分 | 范围查询高效 | 可能数据倾斜 | 中 |
| **地理分片** | 按地域划分 | 就近访问延迟低 | 跨区域查询困难 | 低 |

**分片键选择原则：**
1. 高基数字段（区分度高）
2. 避免热点集中（不要集中到少数分片）
3. 考虑查询模式（范围查询适合范围分片）
4. 考虑扩容频率（频繁扩容选一致性哈希）

## 六、分布式锁（高频考点）

### 6.1 实现原理

**核心命令：**
```bash
SETNX lock_key unique_value    # 互斥
EXPIRE lock_key 30             # 超时释放
```

**合并命令（原子性）：**
```bash
SET lock_key unique_value NX PX 30000
```

---

### 6.2 正确实现

**要求：**
1. 互斥性：同一时刻只有一个客户端持有锁
2. 防死锁：设置超时时间
3. 防误删：只能删除自己的锁
4. 原子性：使用 Lua 脚本

**代码示例：**
```java
// 加锁
String value = UUID.randomUUID().toString();
if (redis.set(lockKey, value, "NX", "PX", 30000) == true) {
    try {
        // 执行临界区代码
    } finally {
        // 解锁（Lua 脚本保证原子性）
        String lua = "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end";
        redis.eval(lua, Collections.singletonList(lockKey), Collections.singletonList(value));
    }
}
```

---

### 6.3 RedLock（Redis 分布式锁）

**场景：**
- Redis 集群环境
- 单点故障可能导致锁失效

**算法步骤：**
1. 客户端获取当前时间 T1
2. 依次向 N 个 Redis 节点申请锁
3. 获取锁成功数量 >= N/2 + 1
4. 计算获取锁耗时 T2 - T1 < 锁有效期
5. 锁成功，否则释放所有锁

## 七、Redis 性能优化

### 7.1 内存优化

| 方法 | 说明 |
|------|------|
| 使用 Hash 代替 String | 对象字段多时用 Hash |

| 压缩列表 | 元素少时用 ziplist |

| 内存淘汰策略 | volatile-lru、allkeys-lru 等 |

---

#### 7.1.1 内存淘汰策略（8 种）

当 Redis 内存达到 `maxmemory` 限制时，需要选择淘汰策略释放空间：

| 策略 | 淘汰范围 | 淘汰算法 | 适用场景 |
|------|---------|---------|---------|
| **noeviction** | 不淘汰 | 写操作返回错误 | 默认，数据不可丢失 |
| **allkeys-lru** | 所有 Key | LRU（最近最少使用） | 通用缓存，推荐 |
| **volatile-lru** | 仅设 TTL 的 Key | LRU | 部分数据需保留 |
| **allkeys-lfu** | 所有 Key | LFU（最少频率使用） | 读热点场景 |
| **volatile-lfu** | 仅设 TTL 的 Key | LFU | 热点+冷数据混合 |
| **allkeys-random** | 所有 Key | 随机淘汰 | 无访问模式偏好 |
| **volatile-random** | 仅设 TTL 的 Key | 随机淘汰 | 随机过期 |
| **volatile-ttl** | 仅设 TTL 的 Key | TTL 最短者优先 | 快速过期场景 |

**LRU vs LFU 对比：**
- LRU（Least Recently Used）：淘汰**最久未被访问**的 Key，适合时间局部性
- LFU（Least Frequently Used）：淘汰**访问频率最低**的 Key，适合热点数据场景

**失效场景（2020年真题考点）：**
- 所有 Key 都设置了 TTL + 使用 `volatile-*` 策略 → 无法淘汰无 TTL 的冷数据
- 内存持续增长说明当前策略无法有效淘汰数据，需要更换为 `allkeys-*` 策略

**配置：**
```conf
maxmemory 2gb
maxmemory-policy allkeys-lru
```

**定义：**
- String > 10KB
- Hash/List/Set/ZSet > 1 万元素

**危害：**

- 内存不均

**解决方案：**

### 例题 4：缓存一致性方案对比（2022年真题改编）

**问题：** 工程师建议实时同步更新缓存和数据库以保证一致性，另一工程师建议异步准实时更新。请描述两种方案的基本思路，并说明仓库管理系统应选择哪种方案。

**答案：**
```
实时同步方案：
- 应用层在写操作时同时更新数据库和缓存（或在事务中完成）
- 保证强一致性，但性能开销大，需分布式事务支持

异步准实时方案：
- 先写数据库，再通过消息队列或 Canal 监听 Binlog 异步更新缓存
- 最终一致性，性能高，适用于读多写少场景

仓库管理系统选择：异步准实时方案
- 理由：仓库管理以读操作为主（库存查询、商品检索），
  写操作不频繁，可容忍秒级数据延迟，异步方案性能更优
```

### 例题 5：一致性哈希 vs 普通哈希（2022年真题改编）

**问题：** 业务增长后需要部署多个 Redis 缓存节点。请简述普通哈希算法和一致性哈希算法的基本原理，并说明为什么推荐一致性哈希。

**答案：**
```
普通哈希：hash(key) % N（N 为节点数）
- 缺点：节点增减时 N 变化，导致绝大部分 Key 重新映射，
  缓存大面积失效，引发缓存重新击穿

一致性哈希：将哈希值映射为 0~2³²-1 的环形空间
- 数据 Key 和节点都映射到环上，Key 顺时针找到第一个节点
- 优点：节点增减时只影响相邻节点，不需要重新哈希全部数据
- 虚拟节点解决数据分布不均问题

选择一致性哈希的原因：扩容影响小，系统稳定性高
```

### 例题 6：布隆过滤器原理（2022年真题改编）

**问题：** 黑客故意发送不存在的商品配送时间查询，导致缓存穿透。请解释布隆过滤器的工作原理和优缺点。

**答案：**
```
原理：
- 位数组（初始全为 0）+ k 个独立哈希函数
- 添加元素：k 个哈希函数计算 k 个位置，将对应位设为 1
- 查询元素：检查 k 个位置，若任一位为 0 → 一定不存在；
  若全为 1 → 可能存在（有假阳性）

核心特性：不存在一定不存在，存在可能存在
- 优点：空间效率极高，查询 O(1)，无假阴性
- 缺点：有假阳性（误判率），不支持删除
- 应用：缓存穿透防护、URL 去重、垃圾邮件过滤
```

### 例题 7：内存淘汰策略失效场景（2020年真题改编）

**问题：** Redis 使用"定期删除 + 惰性删除"策略清理过期 Key。请说明该策略失效的场景，并给出三种内存淘汰机制。

**答案：**
```
失效场景：
- 内存中充满"未过期但不再访问"的冷数据（无 TTL 的冷数据）
- 定期删除和惰性删除都无法处理未过期 Key
- 新数据写入时内存已满，但过期策略无法释放空间

三种内存淘汰机制：
1. allkeys-lru：所有 Key 中按 LRU 淘汰（推荐，通用场景）
2. volatile-lru：仅淘汰设置了 TTL 的 Key 中最近最少使用的
3. volatile-ttl：仅淘汰设置了 TTL 的 Key 中剩余时间最短的
4. allkeys-lfu：所有 Key 中按访问频率最低淘汰
5. noeviction：不淘汰，写操作返回错误（默认策略）
```

### 例题 8：物理过期 vs 逻辑过期对比（2025.11 新考点）

**问题：** 电商系统热点商品缓存击穿，提出两种方案：方案一为互斥锁 + 物理过期，方案二为逻辑过期 + 异步刷新。请从实现复杂度、数据一致性、性能、内存使用、适用场景五个维度对比。

**答案：**
```
物理过期（互斥锁）：
- 实现复杂度：较高（需分布式锁、双重检查）
- 数据一致性：强（数据过期才查 DB）
- 性能：较低（争锁时阻塞等待）
- 内存使用：低（仅存业务数据）
- 适用场景：数据一致性要求高的场景

逻辑过期（异步刷新）：
- 实现复杂度：较低（只需比较时间戳，无需阻塞）
- 数据一致性：弱（可能返回过期数据）
- 性能：高（非阻塞，立即返回旧数据）
- 内存使用：高（每个 Key 多存时间戳字段）
- 适用场景：读多写少、可接受最终一致性的场景
```

### 例题 9：主从全量 + 增量同步（2025.5 新考点）

**问题：** 请描述 Redis 主从全量同步和增量同步的过程。

**答案：**
```
全量同步（首次连接或断线时间过长）：
1. 从库发送 PSYNC 命令到主库
2. 主库执行 BGSAVE 生成 RDB 快照
3. 主库将新写操作写入复制积压缓冲区（repl-backlog-buffer）
4. 主库发送 RDB 快照给从库
5. 从库加载 RDB 恢复数据
6. 主库发送缓冲区中积累的增量写命令
7. 从库执行增量命令，完成同步

增量同步（网络短暂断开后重连）：
1. 从库发送 PSYNC <offset>（携带上次同步的偏移量）
2. 主库检查 offset 是否仍在 repl-backlog-buffer 中
   - 在 → 增量同步：仅补传 offset 之后的写命令
   - 不在 → 全量同步：重新 BGSAVE
3. 主从各自维护 repl-offset 跟踪同步进度
```

## 十、Redis 知识速查卡

### 数据类型一句话记忆

| 类型 | 一句话记忆 | 典型场景 |
|------|-----------|---------|
| String | 计数器、锁、缓存 | 点赞数、分布式锁 |
| List | 双向链表、两端操作 | 消息队列、最新动态 |
| Hash | 键值对、存对象 | 购物车、用户画像 |
| Set | 无序、元素唯一 | 共同好友、标签 |
| ZSet | 有序、带分数 | 排行榜、热搜 |
| HyperLogLog | 基数统计、12KB 固定 | UV 统计 |
| Bitmap | 位图、极致省空间 | 用户签到 |
| Bloom Filter | 不存在一定不存在 | 缓存穿透防护 |

### 三大缓存问题速记

| 问题 | 一句话 | 核心方案 |
|------|--------|---------|
| 穿透 | 查不存在 → 布隆挡刀 + 空值兜底 | Bloom Filter |
| 击穿 | 热点过期 → 互斥锁单飞 + 逻辑过期 | SETNX 锁 |
| 雪崩 | 大量过期 → 随机 TTL + 集群 | 过期时间加随机值 |

### 持久化一句话

| 方案 | 一句话 | 关键词 |
|------|--------|--------|
| RDB | 定时快照，恢复快，可能丢数据 | BGSAVE、dump.rdb |
| AOF | 记录写命令，数据更安全，文件大 | appendfsync、rewrite |
| 混合 | RDB 打底 + AOF 追增量 | Redis 4.0+ |
