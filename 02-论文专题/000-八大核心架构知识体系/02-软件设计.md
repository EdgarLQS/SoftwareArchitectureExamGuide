# 软件设计

## 软件系统划分

### 系统划分层次

| 层次 | 说明 | 产出 |
|------|------|------|
| **系统划分** | 将大系统拆分为子系统 | 子系统边界、接口定义 |
| **模块划分** | 将子系统拆分为模块 | 模块职责、依赖关系 |
| **组件设计** | 模块内部的组件设计 | 组件接口、实现 |

### 子系统划分方法

| 方法 | 说明 | 适用场景 |
|------|------|---------|
| **按功能划分** | 按业务功能划分（用户、订单、支付） | 最常见，职责清晰 |
| **按数据划分** | 按数据域划分（客户数据、产品数据） | 数据驱动系统 |
| **按用户划分** | 按用户角色划分（管理员、普通用户） | 多租户系统 |
| **按业务流程划分** | 按业务流划分（下单→支付→发货） | 流程驱动系统 |

### 模块设计原则

- **高内聚低耦合**：模块内部紧密相关，模块间依赖最小
- **信息隐蔽**：模块内部实现细节对外隐藏
- **模块大小适中**：不宜过大（难以理解）也不宜过小（调用开销大）

### 模块耦合类型（从低到高）

| 类型 | 说明 | 耦合度 |
|------|------|--------|
| **数据耦合** | 模块间通过参数传递简单数据 | 最低 ✅ |
| **标记耦合** | 模块间传递数据结构/记录 | 低 |
| **控制耦合** | 模块间传递控制信息（标志/开关） | 中 |
| **公共耦合** | 模块共享全局数据结构 | 高 |
| **内容耦合** | 一个模块直接访问另一个模块内部 | 最高 ❌ |

### 模块内聚类型（从高到低）

| 类型 | 说明 | 内聚度 |
|------|------|--------|
| **功能内聚** | 模块所有部分共同完成一个功能 | 最高 ✅ |
| **顺序内聚** | 模块处理元素顺序相关（前一个输出是后一个输入） | 高 |
| **通信内聚** | 模块处理元素使用相同输入/输出数据 | 中 |
| **过程内聚** | 模块处理元素按特定顺序执行 | 中 |
| **时间内聚** | 模块处理元素在同一时间段执行（初始化模块） | 低 |
| **逻辑内聚** | 模块通过外部选择执行不同功能 | 低 |
| **巧合内聚** | 模块各部分无逻辑关系 | 最低 ❌ |

## 数据库设计

### 数据库设计六阶段

| 阶段 | 主要任务 | 产出 |
|------|---------|------|
| **需求分析** | 收集数据需求、处理需求 | 数据流图、数据字典 |
| **概念设计** | 抽象实体及关系 | E-R 图 |
| **逻辑设计** | 将 E-R 图转为关系模式 | 关系模式、规范化 |
| **物理设计** | 选择存储结构和存取方法 | 索引设计、分区策略 |
| **实施** | 建库建表、数据导入 | 数据库实例 |
| **运维** | 性能调优、备份恢复 | 监控、优化 |

### E-R 图设计

| 要素 | 说明 | 图形 |
|------|------|------|
| **实体（Entity）** | 现实世界中可区分的对象 | 矩形 |
| **属性（Attribute）** | 实体的特征 | 椭圆 |
| **关系（Relationship）** | 实体之间的联系 | 菱形 |

### 关系模式规范化

| 范式 | 要求 | 解决的问题 |
|------|------|-----------|
| **1NF** | 属性不可再分（原子性） | 消除嵌套 |
| **2NF** | 1NF + 消除非主属性对码的部分依赖 | 消除部分依赖 |
| **3NF** | 2NF + 消除非主属性对码的传递依赖 | 消除传递依赖 |
| **BCNF** | 3NF + 消除主属性对码的传递和部分依赖 | 更严格的 3NF |

### 反规范化（性能优化权衡）

| 方法 | 说明 | 适用场景 |
|------|------|---------|
| **增加冗余列** | 将关联数据冗余存储 | 减少 JOIN 操作 |
| **派生列** | 预先计算并存储计算结果 | 聚合查询频繁 |
| **表拆分** | 大表拆分为小表（水平/垂直） | 单表数据量过大 |
| **宽表** | 将多表合并为一张大表 | 报表/分析场景 |

## 接口设计

### API 设计规范

| 规范 | 说明 |
|------|------|
| **RESTful API** | 基于 HTTP 方法（GET/POST/PUT/DELETE），资源用 URL 表示 |
| **GraphQL** | 客户端按需查询，单次请求获取多种数据 |
| **gRPC** | 基于 Protocol Buffers，高性能 RPC，适合微服务间通信 |

### RESTful API 设计原则

- **资源用名词**：`/users` 而非 `/getUser`
- **HTTP 方法表意图**：GET（查）、POST（增）、PUT（全量改）、PATCH（局部改）、DELETE（删）
- **状态码规范**：200（成功）、201（创建）、400（参数错误）、401（未认证）、403（无权限）、404（不存在）、500（服务器错误）
- **版本管理**：URL 路径版本（`/api/v1/users`）或 Header 版本

### 接口版本管理策略

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **URL 路径** | `/api/v1/resource` | 简单直观 | URL 不优雅 |
| **Header 版本** | `Accept: application/vnd.api.v1+json` | URL 干净 | 不易调试 |
| **查询参数** | `/api/resource?version=1` | 灵活 | 不规范 |

## 基于构件的软件开发 CBSD（2022年论文真题）

### 核心概念

- **构件（Component）**：可复用、自包含、有明确接口的软件单元
- **构件模型**：描述构件的结构和行为（如 EJB、COM+）
- **构件框架**：支持构件组装和运行的基础设施

### CBSD 开发过程

| 步骤 | 任务 | 产出 |
|------|------|------|
| **1. 需求分析** | 确定系统功能和构件需求 | 需求规格 |
| **2. 构件识别** | 识别可复用构件 | 构件清单 |
| **3. 构件获取** | 从构件库获取或自行开发 | 构件实例 |
| **4. 构件适配** | 调整构件接口以适配系统 | 适配代码 |
| **5. 构件组装** | 将构件组合为完整系统 | 系统原型 |
| **6. 系统测试** | 验证系统功能和性能 | 测试报告 |

### CBSD 优缺点

| 优点 | 缺点 |
|------|------|
| 提高开发效率（复用已有构件） | 构件匹配和适配困难 |
| 降低开发成本 | 构件质量难以保证 |
| 缩短开发周期 | 系统灵活性受构件限制 |

## 模型驱动架构 MDA（2024上半年论文真题）

### 核心概念

MDA 是一种以模型为核心的软件开发方法，通过模型转换自动生成代码。

### 三层模型

| 模型 | 全称 | 抽象级别 | 说明 |
|------|------|---------|------|
| **CIM** | Computation Independent Model | 最高 | 业务模型，不涉及计算细节 |
| **PIM** | Platform Independent Model | 中 | 平台无关模型，描述系统结构和行为 |
| **PSM** | Platform Specific Model | 低 | 平台相关模型，针对特定技术平台 |

### MDA 转换过程

```
CIM（业务模型）
  ↓ [PIM 转换]
PIM（平台无关模型，如 UML）
  ↓ [PSM 转换]
PSM（平台相关模型，如 Java EE/.NET）
  ↓ [代码生成]
代码（实现）
```

### MDA 优缺点

| 优点 | 缺点 |
|------|------|
| 提高抽象层次，专注业务逻辑 | 模型转换工具复杂 |
| 平台迁移成本低 | 生成代码质量可能不高 |
| 提高代码一致性 | 学习成本高 |

## UML 建模（关联 2024上半年案例题）

### UML 图分类

| 类别 | 图类型 | 用途 |
|------|--------|------|
| **结构图** | 类图、对象图、组件图、部署图、包图 | 描述系统静态结构 |
| **行为图** | 用例图、序列图、状态图、活动图、通信图 | 描述系统动态行为 |

### 高频 UML 图

| 图 | 核心要素 | 考试重点 |
|----|---------|---------|
| **用例图** | 参与者、用例、关联、包含、扩展 | 需求分析阶段 |
| **类图** | 类、属性、方法、关系（关联/聚合/组合/泛化/实现） | 设计阶段 |
| **序列图** | 生命线、激活条、消息（同步/异步/返回/创建/销毁） | 交互设计 |
| **状态图** | 状态、转换、事件、动作 | 对象生命周期 |
| **活动图** | 活动节点、转换、分支、合并、分叉、汇合 | 业务流程 |

### 类图中的关系

| 关系 | 符号 | 含义 | 耦合度 |
|------|------|------|--------|
| **依赖** | 虚线箭头 | 一个类使用另一个类 | 最低 |
| **关联** | 实线箭头 | 类之间的结构关系 | 低 |
| **聚合** | 空心菱形 | 整体-部分，部分可独立存在 | 中 |
| **组合** | 实心菱形 | 整体-部分，部分不能独立存在 | 高 |
| **泛化** | 空心三角箭头 | 继承关系 | 高 |
| **实现** | 虚线三角箭头 | 实现接口 | 高 |

## 软件架构设计过程

| 阶段 | 任务 | 产出 |
|------|------|------|
| **架构需求分析** | 确定功能需求、质量属性、约束条件 | 架构需求文档 |
| **概念架构设计** | 选择架构风格、划分系统边界 | 概念架构图 |
| **具体架构设计** | 详细设计组件、接口、数据流 | 详细架构图 |
| **架构文档化** | 用标准视图描述架构（4+1 视图） | 架构文档 |
| **架构评审** | 评估架构是否满足需求（ATAM/SAAM） | 评估报告 |

### 4+1 视图模型

| 视图 | 关注点 | 主要图 |
|------|--------|--------|
| **逻辑视图** | 功能需求 | 类图、状态图 |
| **开发视图** | 软件模块组织 | 组件图、包图 |
| **进程视图** | 并发和分布 | 部署图、活动图 |
| **物理视图** | 硬件部署 | 部署图 |
| **场景（+1）** | 用例驱动 | 用例图、序列图 |
