# 架构风格与评估

## 架构风格分类体系

### 数据流风格

| 风格 | 结构 | 适用场景 |
|------|------|---------|
| **管道-过滤器** | 过滤器处理数据流，管道传输 | 编译器、数据处理管线 |
| **批处理序列** | 每个步骤独立处理整个数据集 | 账单生成、报表批处理 |

### 调用/返回风格

| 风格 | 结构 | 适用场景 |
|------|------|---------|
| **主程序/子程序** | 主程序调用子程序，层次化分解 | 传统结构化编程 |
| **面向对象** | 对象封装数据和方法，通过消息通信 | 大多数应用系统 |
| **层次架构** | 系统分层，上层调用下层 | 企业应用、Web 应用 |

### 独立构件风格

| 风格 | 结构 | 适用场景 |
|------|------|---------|
| **通信进程** | 独立进程通过消息通信 | 分布式系统 |
| **事件驱动系统** | 组件通过事件触发执行 | GUI、实时系统、消息驱动 |

### 虚拟机风格

| 风格 | 结构 | 适用场景 |
|------|------|---------|
| **解释器** | 解释执行特定语言 | 脚本引擎、规则引擎 |
| **规则系统** | 基于规则库和推理引擎 | 专家系统、决策支持 |

### 仓库风格

| 风格 | 结构 | 适用场景 |
|------|------|---------|
| **数据库系统** | 中央数据结构 + 独立组件 | 信息系统 |
| **黑板系统** | 共享数据空间 + 多个知识源 | AI 问题求解、语音识别 |
| **超文本系统** | 节点 + 链接 | Wiki、Web |

## 分层架构

| 维度 | 说明 |
|------|------|
| **典型分层** | 表现层 → 业务逻辑层 → 数据访问层 → 数据库 |
| **优点** | 职责分离、可维护性强、可测试性强、团队可并行开发 |
| **缺点** | 性能损耗（逐层调用）、可能导致过度设计、跨层调用困难 |
| **变体** | 开放层架构（允许跳层调用）、反向依赖（下层依赖上层接口） |
| **适用** | 传统企业应用、中小型 Web 应用 |

## 微服务架构（2021年论文真题）

### 核心特征

- **服务即组件**：每个服务是独立可替换的组件
- **围绕业务能力组织**：按业务领域而非技术分层
- **去中心化治理**：技术异构、独立技术栈选择
- **去中心化数据管理**：每个服务独立数据库
- **基础设施自动化**：CI/CD、容器化、编排
- **容错设计**：熔断、降级、隔离

### 服务拆分原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 一个服务只做一件事 |
| **DDD 领域驱动** | 按限界上下文（Bounded Context）拆分 |
| **高内聚低耦合** | 服务内部高度相关，服务间依赖最小 |
| **康威定律** | 架构与组织沟通结构保持一致 |
| **数据自治** | 服务拥有自己的数据存储，不共享数据库 |

### 服务通信

| 方式 | 技术 | 特点 |
|------|------|------|
| **同步** | REST/HTTP、gRPC | 简单直接、阻塞调用 |
| **异步** | 消息队列（Kafka/RabbitMQ） | 解耦、削峰、最终一致 |

### 服务治理

| 能力 | 说明 | 代表技术 |
|------|------|---------|
| **服务注册与发现** | 服务上线/下线自动感知 | Consul、Nacos、Eureka |
| **负载均衡** | 请求均匀分配到服务实例 | Ribbon、Nginx、K8s Service |
| **熔断降级** | 故障隔离与快速失败 | Hystrix、Sentinel、Resilience4j |
| **链路追踪** | 请求全链路追踪 | Jaeger、Zipkin、SkyWalking |
| **配置中心** | 配置集中管理、动态刷新 | Nacos、Apollo、Spring Cloud Config |
| **API 网关** | 统一入口、路由、鉴权、限流 | Kong、Zuul、APISIX |

### 微服务 vs 单体 vs SOA

| 维度 | 单体架构 | 微服务架构 | SOA |
|------|---------|-----------|-----|
| 服务粒度 | 无服务概念 | 细粒度 | 粗粒度 |
| 数据管理 | 共享数据库 | 独立数据库 | 共享 ESB |
| 通信方式 | 方法调用 | REST/gRPC/消息 | SOAP/ESB |
| 部署方式 | 整体部署 | 独立部署 | 集中部署 |
| 技术栈 | 单一技术栈 | 技术异构 | 技术异构 |
| 治理方式 | 中心化 | 去中心化 | ESB 中心化 |
| 适用规模 | 小型团队/项目 | 中大型团队 | 企业级集成 |

## Serverless 架构（2025下半年论文真题）

### 核心组成

| 组件 | 说明 | 代表产品 |
|------|------|---------|
| **FaaS（函数即服务）** | 事件触发的无状态函数执行 | AWS Lambda、阿里云函数计算 |
| **BaaS（后端即服务）** | 托管的后端服务（数据库、认证、存储） | Firebase、Auth0 |

### 架构特征

- **无服务器管理**：开发者不关心基础设施
- **事件驱动**：由 HTTP 请求、消息队列、定时器等触发
- **自动弹性伸缩**：按需分配资源
- **按使用量计费**：按执行次数和时间计费
- **无状态**：函数执行完即释放，状态需外部存储

### Serverless vs 微服务 vs 传统架构

| 维度 | 传统架构 | 微服务架构 | Serverless |
|------|---------|-----------|-----------|
| 运维复杂度 | 高（需管理服务器） | 高（需编排调度） | 低（平台托管） |
| 启动速度 | 慢 | 中 | 慢（冷启动问题） |
| 成本控制 | 预留资源，利用率低 | 容器化提升利用率 | 按使用计费，最经济 |
| 适合场景 | 稳定负载、长运行任务 | 中大型业务系统 | 突发流量、事件处理 |

### 冷启动问题

- **原因**：函数长时间未调用后需重新初始化环境
- **影响**：首次请求延迟显著增加（秒级）
- **解决方案**：定时保活调用、预置实例、选择启动快的运行时

## 事件驱动架构 EDA（2025上半年论文真题）

### 核心组件

| 组件 | 职责 |
|------|------|
| **事件生产者** | 产生事件，发布到事件通道 |
| **事件消费者** | 订阅并处理事件 |
| **事件通道** | 传输事件的中介（消息队列、事件总线） |

### 事件模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **发布-订阅** | 生产者发布事件，多个消费者订阅 | 消息通知、数据同步 |
| **事件通知** | 状态变更时发送通知 | 审计日志、告警 |
| **事件溯源** | 将状态变更作为事件序列持久化 | 金融交易、审计系统 |

### CQRS + 事件溯源

| 概念 | 说明 |
|------|------|
| **CQRS（命令查询职责分离）** | 将写操作（Command）和读操作（Query）分离为不同模型 |
| **事件溯源** | 系统状态由事件序列推导，事件为唯一事实源 |
| **结合优势** | 写入用事件溯源保证一致性，读取用独立模型优化性能 |

### EDA 优缺点

| 优点 | 缺点 |
|------|------|
| 松耦合（生产者不知消费者） | 调试困难（调用链路隐式） |
| 高扩展性（消费者独立扩展） | 数据一致性复杂（最终一致） |
| 高可用（组件故障不影响其他） | 运维复杂度高 |
| 天然支持异步和削峰 | 事件版本管理困难 |

## SOA 面向服务架构（2023/2024下半年论文真题）

### 核心概念

- **服务**：自包含、可复用的业务功能单元
- **服务契约**：服务接口定义（WSDL）
- **服务注册**：服务目录（UDDI）
- **服务编排**：多个服务组合成业务流程（BPEL）

### WebService 技术栈

| 技术 | 作用 |
|------|------|
| **SOAP** | 消息传输协议（XML 格式） |
| **WSDL** | 服务描述语言（接口定义） |
| **UDDI** | 服务注册与发现 |

### SOA 与微服务对比

见上文"微服务 vs 单体 vs SOA"表格。

## 云原生架构（2020/2025下半年论文真题）

### 四大核心要素

| 要素 | 说明 | 代表技术 |
|------|------|---------|
| **容器化** | 应用打包与运行标准化 | Docker、containerd |
| **微服务** | 细粒度服务拆分 | Spring Cloud、Dubbo |

| Serverless | AWS Lambda、Knative |

## 湖仓一体架构（2022年论文真题）

### 概念

数据湖（灵活存储原始数据）与数据仓库（结构化分析数据）的融合架构。

### 四类关键特征

| 特征 | 说明 |
|------|------|
| **事务一致性** | 支持 ACID 事务，保证数据一致性 |
| **高并发实时处理** | 支持高并发查询和实时数据写入 |
| **多元数据分析** | 支持 BI、机器学习、即席查询等多种分析 |
| **统一数据存储** | 一套存储同时支持结构化和非结构化数据 |

### 数据湖 vs 数据仓库 vs 湖仓一体

| 维度 | 数据湖 | 数据仓库 | 湖仓一体 |
|------|--------|---------|---------|
| 数据类型 | 原始数据（结构化+非结构化） | 结构化处理数据 | 统一存储 |
| 使用者 | 数据科学家 | 业务分析师 | 全员 |
| 灵活性 | 高（Schema-on-Read） | 低（Schema-on-Write） | 中 |
| 事务支持 | 弱 | 强 | 强 |
| 成本 | 低 | 高 | 中 |
| 代表技术 | HDFS、S3 | Redshift、ClickHouse | Delta Lake、Iceberg、Hudi |

## 区块链架构（2022年论文真题）

### 核心特征

- **去中心化**：无中心化控制点
- **分布式账本**：所有节点共享同一份数据
- **共识机制**：PoW、PoS、PBFT 等保证一致性
- **智能合约**：自动执行的合约逻辑
- **不可篡改**：链式结构保证数据历史不可修改

### 区块链分类

| 类型 | 准入 | 共识 | 适用场景 |
|------|------|------|---------|
| **公有链** | 任何人 | PoW/PoS | 加密货币 |
| **联盟链** | 授权组织 | PBFT/Raft | 企业间协作 |
| **私有链** | 单一组织 | 集中式 | 企业内部 |

## ATAM 架构评估方法

### ATAM 完整流程（4 阶段 12 步骤）

#### 阶段 1：准备与介绍（步骤 1-3）

1. **介绍 ATAM 方法**：向参与方说明评估流程
2. **介绍业务驱动因素**：项目背景、业务目标、约束条件
3. **介绍系统架构**：架构视图、关键技术决策

#### 阶段 2：调查与分析（步骤 4-6）

4. **确定架构方法**：识别架构策略和模式
5. **生成质量属性效用树**：构建效用树（Utility Tree），排列质量属性优先级
6. **分析架构方法**：针对高风险场景分析架构决策

#### 阶段 3：测试与验证（步骤 7-9）

7. **从效用树中选取场景**：选择优先级最高的场景进行深入分析
8. **分析架构方法（续）**：针对选定场景验证架构是否满足需求
9. **描述场景对架构的影响**：标注受场景影响的架构组件

#### 阶段 4：报告与改进（步骤 10-12）

10. **分组深入分析**：分小组对不同场景进行深入分析
11. **汇总分析结果**：各组汇报，形成完整分析
12. **生成 ATAM 报告**：总结风险点、敏感点、权衡点、非风险点

### ATAM 核心概念扩展

| 概念 | 说明 | 示例 |
|------|------|------|
| **风险点（Risk）** | 可能导致质量属性无法满足的架构决策 | 单点故障、未做负载均衡 |
| **非风险点（Non-Risk）** | 经分析确认没有问题的架构决策 | 已做数据库主从 |
| **敏感点** | 某参数的微小变化会显著影响某个质量属性 | 加密密钥长度影响安全性 |
| **权衡点** | 同时影响多个质量属性（通常一正一负） | 加密提升安全但降低性能 |

### 质量属性场景六要素

| 要素 | 说明 | 示例（性能场景） |
|------|------|----------------|
| **刺激源（Source）** | 产生刺激的实体 | 用户提交请求 |
| **刺激（Stimulus）** | 到达系统的事件 | 发送查询请求 |
| **制品（Artifact）** | 被刺激作用的系统部分 | Web 服务器 |
| **环境（Environment）** | 刺激发生时的条件 | 正常负载运行中 |
| **响应（Response）** | 系统的反应 | 返回查询结果 |
| **响应度量（Measure）** | 对响应的评估标准 | 响应时间 < 2 秒 |

### ATAM vs SAAM vs ARID

| 方法 | 全称 | 关注点 | 适用场景 |
|------|------|--------|---------|
| **ATAM** | Architecture Tradeoff Analysis Method | 质量属性权衡 | 综合评估，最常用 |
| **SAAM** | Software Architecture Analysis Method | 可修改性分析 | 评估架构的可扩展性 |
| **ARID** | Active Reviews for Intermediate Designs | 设计评审 | 中间设计阶段评审 |

## 六大质量属性

| 质量属性 | 定义 | 关注点 | 常用战术 |
|---------|------|--------|---------|
| **性能** | 系统响应能力 | 响应时间、吞吐量 | 并发、缓存、资源池 |
| **可用性** | 系统正常运行能力 | 故障检测与恢复 | 心跳、冗余、投票 |
| **可修改性** | 系统变更的容易程度 | 扩展/修改成本 | 低耦合、抽象、插件化 |
| **安全性** | 防止非法访问和破坏 | 认证、授权、保密 | 审计、加密、最小权限 |
| **可测试性** | 发现和定位缺陷的容易程度 | 测试覆盖、隔离 | 依赖注入、Mock、日志 |
| **易用性** | 用户使用的方便程度 | 学习成本、操作效率 | 一致性、反馈、容错 |

## 架构风格选型指南

| 场景 | 推荐架构 | 理由 |
|------|---------|------|
| 小型企业内部系统 | 分层架构 | 简单、易维护 |
| 大型互联网应用 | 微服务架构 | 独立部署、弹性扩展 |
| 实时事件处理 | 事件驱动架构 | 松耦合、高响应 |
| 企业系统集成 | SOA + ESB | 统一总线集成 |
| 快速原型/突发流量 | Serverless | 免运维、按需付费 |
| 大数据全量分析 | 湖仓一体 | 统一存储多元分析 |
| 去中心化信任场景 | 区块链 | 不可篡改、共识机制 |
