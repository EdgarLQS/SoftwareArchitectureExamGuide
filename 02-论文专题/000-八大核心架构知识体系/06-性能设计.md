# 性能设计

## 缓存技术

### 缓存层次体系

| 层次 | 技术 | 特点 | 适用场景 |
|------|------|------|---------|
| **浏览器缓存** | HTTP Cache-Control、ETag | 离用户最近，减少网络请求 | 静态资源、API 响应 |
| **CDN 缓存** | 边缘节点缓存 | 地理就近，减轻源站压力 | 图片、视频、静态页面 |
| **应用层缓存** | Redis、Memcached、Caffeine | 进程内/外缓存，毫秒级响应 | 热点数据、会话存储 |
| **数据库缓存** | 查询缓存、缓冲池（Buffer Pool） | 数据库内置，自动管理 | 频繁查询结果 |

### 缓存模式

| 模式 | 流程 | 适用场景 |
|------|------|---------|
| **Cache Aside（旁路缓存）** | 读：先读缓存，miss 则读 DB 并写入缓存；写：先更新 DB，再删除缓存 | 最常用，Redis 默认推荐 |
| **Read Through（读穿透）** | 缓存负责从 DB 加载数据，应用只与缓存交互 | 缓存与 DB 解耦 |
| **Write Through（写穿透）** | 写缓存时同步写 DB | 强一致性要求 |
| **Write Behind（写回）** | 只写缓存，异步批量刷盘 | 高吞吐写入场景 |

### 缓存三大问题及解决方案

| 问题 | 现象 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 查询不存在的数据，每次 miss 打到 DB | 布隆过滤器、缓存空值 |
| **缓存击穿** | 热点 key 过期瞬间，大量请求打到 DB | 互斥锁（SETNX）、热点 key 永不过期 |
| **缓存雪崩** | 大量 key 同时过期或 Redis 宕机 | 过期时间加随机值、集群高可用、限流降级 |

### 缓存一致性策略

| 策略 | 说明 | 一致性强度 | 性能影响 |
|------|------|-----------|---------|
| **过期淘汰** | 设置 TTL，过期后重新加载 | 弱（最终一致） | 最低 |
| **主动删除** | 更新 DB 后主动删除缓存 | 中 | 低 |
| **订阅 Binlog** | 监听 DB Binlog 自动更新缓存 | 强 | 中（需 Canal 等工具） |
| **双写同步** | 同时写 DB 和缓存 | 最强 | 高（需分布式事务） |

## 数据分片技术（2020年论文真题）

### 分片方式对比

| 方式 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **水平分片** | 按行拆分，不同行存储到不同节点 | 单表数据量降低、写入性能提升 | 跨片查询困难 | 用户表、订单表 |
| **垂直分片** | 按列拆分，不同字段存储到不同节点 | 减少 IO 竞争、热点字段隔离 | 需要 JOIN 查询 | 大字段分离、冷热分离 |
| **分库** | 不同数据库实例 | 提升并发处理能力 | 跨库事务复杂 | 高并发写入 |

### 分片策略

| 策略 | 原理 | 优点 | 缺点 | 扩容难度 |
|------|------|------|------|---------|
| **Hash 分片** | 对分片键取 Hash 后取模 | 数据分布均匀 | 扩容需全量重分配 | 高（需数据迁移） |
| **一致性 Hash** | Hash 环 + 虚拟节点 | 扩容只影响相邻节点 | 实现复杂 | 低 |
| **范围分片** | 按区间划分（如用户 ID 段） | 范围查询高效 | 可能数据倾斜 | 中（追加新片） |
| **地理分片** | 按地域划分数据 | 就近访问延迟低 | 跨区域查询困难 | 低 |

### 分片键选择原则

- **高频查询字段**：分片键应是 WHERE 条件中最常用的字段
- **均匀分布**：分片键的值应均匀分布，避免数据倾斜
- **避免跨片 JOIN**：相关业务数据使用相同分片键
- **考虑扩容**：选择未来不太会改变的分片维度

### 跨片查询处理

| 方案 | 说明 |
|------|------|
| **全局表** | 小表全片复制（如字典表） |
| **绑定表** | 关联表使用相同分片键，保证同片 |
| **共享查询** | 路由到所有片查询后聚合结果 |
| **异构索引库** | 将分片数据同步到 ES/ClickHouse 等分析引擎 |

## 数据库性能优化

### 索引设计

| 索引类型 | 说明 | 适用场景 |
|---------|------|---------|
| **B+ 树索引** | MySQL 默认索引，支持范围查询 | 等值查询、范围查询、排序 |
| **Hash 索引** | 等值查询 O(1)，不支持范围 | 精确匹配 |
| **联合索引** | 多列组合索引，最左前缀匹配 | 多条件查询 |
| **覆盖索引** | 索引包含查询所需全部字段 | 避免回表 |

### 读写分离

| 方案 | 说明 | 延迟问题 |
|------|------|---------|
| **主从复制** | 主库写，从库读，异步复制 | 存在复制延迟 |
| **半同步复制** | 至少一个从库确认后才返回 | 延迟降低但影响写性能 |
| **读写路由** | 代理层（MyCat/ShardingSphere）自动路由 | 读请求分散到多个从库 |

### SQL 优化

- **EXPLAIN 分析执行计划**：关注 type（访问类型）、rows（扫描行数）、Extra（额外信息）
- **避免全表扫描**：合理使用索引、LIMIT 分页
- **避免 SELECT \***：只查询需要的字段
- **批量操作**：批量 INSERT/UPDATE 减少网络往返

## 异步处理

- **消息队列**：Kafka（高吞吐日志）、RabbitMQ（灵活路由）、RocketMQ（事务消息）
- **事件驱动**：组件通过事件触发执行，天然支持异步和削峰
- **后台任务**：定时任务、异步通知、离线计算

## 负载均衡（2025上半年论文真题）

### 负载均衡层级

| 层级 | 技术 | 特点 |
|------|------|------|
| **DNS 层** | DNS 轮询、DNS 解析就近 | 粗粒度，受 DNS 缓存影响 |
| **硬件层** | F5 BIG-IP | 高性能、高成本 |
| **四层（传输层）** | LVS、Nginx Stream | 基于 IP+端口转发 |
| **七层（应用层）** | Nginx、HAProxy | 基于 URL/Header 路由 |
| **云原生层** | K8s Ingress、Service Mesh | 动态感知、自动编排 |

### 负载均衡策略

#### 静态策略

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| **轮询（Round Robin）** | 依次分配请求 | 服务器性能相近 |
| **加权轮询** | 按权重比例分配 | 服务器性能不同 |
| **哈希** | 对客户端 IP/Session 取哈希 | 会话保持 |
| **随机** | 随机选择后端 | 简单场景 |

#### 动态策略

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| **最小连接数** | 分配给当前连接数最少的后端 | 请求处理时长差异大 |
| **最快响应时间** | 分配给响应最快的后端 | 对延迟敏感 |
| **负载反馈** | 根据后端 CPU/内存负载动态分配 | 基础设施感知 |

### 负载均衡与论文写作

在论文中论述负载均衡设计时应包含：
1. **业务背景**：为什么需要负载均衡（流量增长、单点瓶颈）
2. **选型理由**：为什么选择某种策略和实现
3. **具体实施**：配置参数、权重设置、健康检查机制
4. **效果验证**：QPS 提升、响应时间降低、可用性提升

## 限流与熔断（2025下半年秒杀真题关联）

### 限流算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **计数器** | 固定时间窗口内计数 | 实现简单 | 窗口边界突刺 |
| **滑动窗口** | 多个子窗口滑动统计 | 解决边界突刺 | 内存开销较大 |
| **令牌桶** | 固定速率生成令牌，请求需获取令牌 | 允许突发流量（桶容量） | 实现稍复杂 |
| **漏桶** | 固定速率流出请求 | 流量平滑 | 无法应对突发 |

### 限流维度

| 维度 | 说明 | 实现位置 |
|------|------|---------|
| **网关限流** | 统一入口限流 | API 网关（Kong/APISIX） |
| **应用限流** | 服务级别限流 | 应用中间件（Sentinel） |
| **接口限流** | 细粒度到单个 API | 代码层注解 |
| **用户限流** | 针对单个用户/IP | 令牌桶 per user |

### 熔断器模式（Circuit Breaker）

| 状态 | 行为 | 转换条件 |
|------|------|---------|
| **Closed（关闭）** | 正常通过请求 | 失败率超过阈值 → Open |
| **Open（打开）** | 拒绝所有请求，快速失败 | 等待超时 → Half-Open |
| **Half-Open（半开）** | 放行少量试探请求 | 成功 → Closed；失败 → Open |

### 服务降级策略

| 策略 | 说明 | 示例 |
|------|------|------|
| **默认值降级** | 返回默认值或缓存数据 | 商品详情返回缓存快照 |
| **功能降级** | 关闭非核心功能 | 秒杀期间关闭评论、推荐 |
| **延迟处理** | 请求入队后异步处理 | 下单后排队处理 |
| **限流排队** | 超出容量时排队等待 | 秒杀请求进入等待队列 |

## 性能测试（2025下半年论文真题）

### 测试类型

| 类型 | 目的 | 关键问题 |
|------|------|---------|
| **负载测试** | 测量正常负载下的性能 | 系统在预期负载下表现如何？ |
| **压力测试** | 逐步增加负载直到系统崩溃 | 系统极限在哪里？ |
| **并发测试** | 测试大量用户同时操作 | 并发用户数是否影响响应？ |
| **稳定性测试** | 长时间运行观察系统状态 | 是否有内存泄漏、性能衰减？ |
| **峰值测试** | 模拟突发流量高峰 | 秒杀/抢购场景的承载能力 |

### 关键性能指标

| 指标 | 说明 | 计算公式/参考值 |
|------|------|----------------|
| **响应时间（RT）** | 从请求发出到收到响应的时间 | P50、P90、P99 分位 |
| **TPS** | 每秒事务处理数 | 事务数 / 测试时间 |
| **QPS** | 每秒查询数 | 查询数 / 测试时间 |
| **并发用户数** | 系统同时承载的用户数 | 虚拟用户数 |
| **资源利用率** | CPU、内存、磁盘 IO、网络 IO | CPU < 70% 为健康 |
| **错误率** | 失败请求占比 | 错误数 / 总请求数 < 1% |

### 性能测试流程

1. **需求分析**：确定性能目标（RT < 2s，TPS > 1000）
2. **测试设计**：设计测试场景、构造测试数据
3. **环境准备**：搭建与生产一致的性能测试环境
4. **执行测试**：逐步加压，记录各指标
5. **结果分析**：定位瓶颈（CPU/IO/DB/代码）
6. **优化迭代**：针对性优化后重新测试

### 常用工具

| 工具 | 特点 | 适用场景 |
|------|------|---------|
| **JMeter** | 开源、图形化、插件丰富 | 接口性能测试 |

## 秒杀/高并发场景架构（2025下半年论文真题）

### 核心技术挑战

| 挑战 | 说明 |
|------|------|
| **瞬时高并发** | 短时间内大量请求涌入 |
| **库存超卖** | 并发扣减库存导致数据不一致 |
| **数据库瓶颈** | 大量读写操作导致数据库崩溃 |
| **黄牛/刷单** | 恶意请求抢占资源 |

### "分流-提速-减压-兜底"设计框架

| 阶段 | 技术手段 | 效果 |
|------|---------|------|
| **分流** | CDN 静态资源、DNS 负载均衡、网关层限流 | 过滤无效请求 |
| **提速** | 多级缓存（浏览器/CDN/Redis/本地）、静态化页面 | 加速响应 |
| **减压** | 消息队列异步处理、库存预扣减、读写分离 | 降低 DB 压力 |
| **兜底** | 熔断降级、服务降级、排队等待、错误页 | 保证系统不崩溃 |

### 防超卖方案

| 方案 | 原理 | 适用并发量 |
|------|------|-----------|
| **数据库乐观锁** | UPDATE ... WHERE stock = old_stock | 低并发 |
| **Redis 原子操作** | DECR 原子递减 | 中高并发 |
| **Redis + Lua 脚本** | 检查库存 + 扣减在脚本中原子执行 | 高并发（推荐） |
| **消息队列串行化** | 请求入队，单线程消费扣减库存 | 极高并发 |

### 论文写作要点

在秒杀场景论文中应论述：
1. **项目背景**：秒杀活动的业务价值和技术挑战
2. **架构设计**：整体架构图，分层设计
3. **关键技术**：缓存策略、限流算法、库存防超卖
4. **性能数据**：压测结果对比（优化前 vs 优化后）
5. **经验总结**：遇到的问题和解决过程
