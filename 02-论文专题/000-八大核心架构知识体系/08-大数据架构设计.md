# 大数据架构设计

## Lambda 架构（2024上半年论文真题）

### 三层架构详解

```
                    ┌──────────────────────────────────────┐
                    │             服务层（Serving）          │
                    │    合并批处理和实时处理的结果           │
                    └──────────────┬───────────────────────┘
                                   │
              ┌────────────────────┴────────────────────┐
              │                                         │
┌─────────────────────────┐        ┌────────────────────┐
│   批处理层（Batch）       │        │  速度层（Speed）     │
│   处理全量历史数据         │        │  处理增量实时数据     │
│   Hadoop / Spark         │        │  Storm / Flink      │
└────────────┬──────────────┘        └────────┬───────────┘
             │                                │
             └────────────────┬───────────────┘
                              │
                    ┌─────────┴─────────┐
                    │   数据源 → Kafka   │
                    └───────────────────┘
```

| 层次 | 职责 | 技术选型 | 特点 |
|------|------|---------|------|
| **批处理层** | 处理全量历史数据，生成批处理视图 | Hadoop MapReduce、Spark | 高延迟（小时级）、高精度 |
| **速度层** | 处理增量数据，弥补批处理延迟 | Storm、Flink、Spark Streaming | 低延迟（秒级）、近似精度 |
| **服务层** | 合并两层结果，提供统一查询接口 | Druid、Elasticsearch、HBase | 统一视图、快速查询 |

### Lambda 架构优缺点

| 优点 | 缺点 |
|------|------|
| 容错性强（批处理层可重算） | 维护两套代码（批+流），运维成本高 |
| 支持历史数据分析 | 数据合并逻辑复杂 |
| 兼顾精度和实时性 | 服务层需合并两种视图 |

## Kappa 架构

### 核心理念

Kappa 架构是对 Lambda 的简化——**只用一套流处理管道**，摒弃批处理层。所有数据（历史和实时）都通过流处理引擎处理。

```
数据源 → Kafka（日志存储，保留全量历史）→ Flink 流处理 → 服务层 → 查询
```

| 特性 | 说明 |
|------|------|
| **统一管道** | 只有一套流处理代码，降低维护成本 |
| **日志回溯** | 需要重新处理时，从 Kafka 重放历史数据 |
| **技术简化** | 不再需要批处理框架 |

### Kappa vs Lambda

| 维度 | Lambda | Kappa |
|------|--------|-------|
| 处理管道 | 两套（批+流） | 一套（流） |
| 代码复杂度 | 高（维护两套逻辑） | 低（一套代码） |
| 历史数据处理 | 批处理框架 | Kafka 重放 + 流处理 |
| 运维成本 | 高 | 低 |
| 适用场景 | 精度要求极高的离线分析 | 实时性优先、事件驱动 |

## 湖仓一体架构（2022年论文真题）

### 概念

数据湖（灵活存储原始数据）与数据仓库（结构化分析数据）的融合架构，实现一套存储同时支持结构化和非结构化数据。

### 四类关键特征

| 特征 | 说明 |
|------|------|
| **事务一致性** | 支持 ACID 事务，保证读写一致性 |
| **高并发实时处理** | 支持高并发查询和实时数据写入 |
| **多元数据分析** | 同时支持 BI、机器学习、即席查询等多种分析 |
| **统一数据存储** | 一套存储同时支持结构化和非结构化数据 |

### 数据湖 vs 数据仓库 vs 湖仓一体

| 维度 | 数据湖 | 数据仓库 | 湖仓一体 |
|------|--------|---------|---------|
| **数据类型** | 原始数据（结构化+半结构化+非结构化） | 结构化处理后数据 | 统一存储 |
| **写入模式** | Schema-on-Read（读时定义） | Schema-on-Write（写时定义） | 灵活支持 |
| **使用者** | 数据科学家 | 业务分析师 | 全员 |
| **事务支持** | 弱或无 | 强 ACID | 强 ACID |
| **查询性能** | 低（需全表扫描） | 高（优化索引） | 中高 |
| **成本** | 低（对象存储） | 高（专有硬件） | 中 |
| **代表技术** | HDFS、S3 | Redshift、ClickHouse | Delta Lake、Iceberg、Hudi |

### 湖仓一体技术实现

| 技术 | 特点 | 开发方 |
|------|------|--------|
| **Delta Lake** | ACID 事务、时间旅行、Schema 演进 | Databricks |
| **Apache Iceberg** | 隐藏分区、模式演进、多引擎兼容 | Netflix |
| **Apache Hudi** | 增量处理、Upsert 支持、近实时 | Uber |

### 湖仓一体优势

- **统一数据平台**：消除数据孤岛，一份数据多种用途
- **降低运维成本**：不再维护两套系统
- **支持 AI/ML**：原始数据直接用于模型训练
- **灵活扩展**：按需扩展计算和存储

## 数据湖

### 数据湖核心组件

| 组件 | 职责 | 代表技术 |
|------|------|---------|
| **数据存储** | 存储原始数据（结构化+非结构化） | HDFS、Amazon S3 |
| **数据摄取** | 批量/实时数据导入 | Sqoop、Flume、Kafka |
| **数据治理** | 元数据管理、数据目录、权限 | Apache Atlas |
| **数据处理** | 批处理/流处理/交互式查询 | Spark、Flink、Presto |
| **数据服务** | 数据 API、BI 工具对接 | JDBC/ODBC、REST API |

### 数据治理挑战

| 挑战 | 说明 | 解决方案 |
|------|------|---------|
| **数据沼泽** | 缺乏治理导致数据不可用 | 元数据管理、数据目录 |
| **数据质量** | 原始数据未清洗 | 数据质量管理流程 |
| **安全合规** | 敏感数据存储 | 数据脱敏、加密、访问控制 |
| **成本失控** | 无限存储导致成本上升 | 生命周期管理、冷热分层 |

## 多模型数据管理（2025上半年论文真题）

### 数据模型类型

| 模型类型 | 数据结构 | 代表数据库 | 适用场景 |
|---------|---------|-----------|---------|
| **关系模型** | 表、行、列 | MySQL、PostgreSQL | 事务处理、报表 |
| **文档模型** | JSON/BSON 文档 | MongoDB、CouchDB | 灵活 Schema、内容管理 |
| **图模型** | 节点、边、属性 | Neo4j、JanusGraph | 社交网络、推荐、知识图谱 |
| **键值模型** | Key-Value 对 | Redis、DynamoDB | 缓存、会话、配置 |
| **时序模型** | 时间序列数据 | InfluxDB、TimescaleDB | 监控、IoT、日志 |

### 多模型数据库

| 数据库 | 支持的模型 | 特点 |
|--------|-----------|------|
| **ArangoDB** | 文档 + 图 + 键值 | 统一查询语言 AQL |
| **OrientDB** | 文档 + 图 + 对象 | 原生图 + 文档融合 |
| **Azure Cosmos DB** | 文档 + 图 + 列族 + 键值 | 全球分布式 |
| **PostgreSQL + 插件** | 关系 + JSON + 图（AGE） | 开源生态丰富 |

### 多模型数据管理的技术优势

| 优势 | 说明 |
|------|------|
| **数据一致性** | 一套系统处理多种模型，避免跨系统同步 |
| **开发效率** | 统一 API 和查询语言 |
| **系统扩展性** | 按需扩展不同模型的能力 |
| **降低运维** | 减少数据库种类，降低 DBA 负担 |

## 大数据处理技术

### 批处理

| 技术 | 特点 | 适用场景 |
|------|------|---------|
| **Hadoop MapReduce** | 基于磁盘，高容错，高延迟 | 海量离线数据分析 |
| **Apache Spark** | 基于内存，比 MR 快 10-100 倍 | 迭代计算、机器学习 |

### 流处理

| 技术 | 特点 | 适用场景 |
|------|------|---------|
| **Apache Storm** | 低延迟（毫秒级），逐条处理 | 实时告警、实时计算 |
| **Spark Streaming** | 微批处理，秒级延迟 | 实时统计 |
| **Apache Flink** | 真正的流处理，Exactly-Once，状态管理 | 实时数仓、CEP、CDC（最主流） |

### OLAP 引擎

| 引擎 | 特点 | 适用场景 |
|------|------|---------|
| **ClickHouse** | 列式存储，极速查询 | 日志分析、指标查询 |
| **Druid** | 实时摄入，亚秒查询 | 交互式分析看板 |
| **Apache Kylin** | 预计算 Cube | 固定维度 BI 查询 |

## 数据集成与 ETL 工具链

| 环节 | 工具 | 说明 |
|------|------|------|
| **数据采集** | Flume、Logstash、Sqoop | 日志采集、关系库导入 |
| **数据传输** | Kafka、Pulsar | 高吞吐消息队列 |
| **数据转换** | Spark、Flink | 批量/流式数据转换 |
| **数据调度** | Airflow、DolphinScheduler | 工作流编排 |
| **数据存储** | HDFS、S3、HBase | 分布式存储 |

## 大数据架构选型指南

| 场景 | 推荐架构 | 理由 |
|------|---------|------|
| 离线数据分析 | Lambda（批处理层） | 精度高，成本低 |
| 实时数据监控 | Kappa / Flink 流处理 | 延迟低，运维简单 |
| 多元数据分析 | 湖仓一体 | 统一存储，BI+ML 兼顾 |
| 历史数据仓库 | 数据仓库 + ETL | 结构化分析，SQL 友好 |
| AI/ML 数据平台 | 数据湖 | 原始数据直接用于训练 |
| 实时+离线兼顾 | Lambda | 批流结合 |
| 简化运维 | Kappa | 一套代码，易于维护 |

## 论文写作要点

在大数据架构相关论文中应论述：
1. **数据规模**：数据量、增长速率、数据类型
2. **架构选型理由**：为什么选择 Lambda/Kappa/湖仓一体
3. **技术栈选择**：各层的技术选型及理由
4. **数据处理流程**：从采集到查询的完整链路
5. **性能数据**：处理延迟、吞吐量、存储成本
6. **经验教训**：数据质量、运维挑战、成本优化
