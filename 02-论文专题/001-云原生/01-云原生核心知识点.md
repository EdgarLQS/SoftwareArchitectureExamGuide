## 一、云原生四大支柱

### 1. 微服务（Microservices）

> 微服务是云原生四大支柱之一。关于微服务拆分、DDD、服务治理、熔断降级、链路追踪等详细内容，请参考 `001-微服务/01-微服务核心知识点.md`。

**核心概念：** 将大型应用拆分为小型独立服务，每个服务可独立部署、独立伸缩、独立故障隔离。

---

### 2. 容器化（Containerization）

**核心概念：**
- 如果把微服务比作"小盒子"，容器化就是把这些小盒子打包成标准"集装箱"
- 容器提供一致的运行环境，消除"在我机器上能跑"的问题

**容器化的价值：**
- 环境一致性：开发、测试、生产环境完全一致
- 快速启动：秒级完成新实例启动
- 资源隔离：每个容器拥有独立的资源视图
- 便携性：可在任何支持容器的平台上运行

---

### 3. DevOps（开发运维一体化）

**思考题：如果你有 1000 个容器需要手动一个一个更新，最让你头疼的问题是什么？**

**答案：效率和一致性**
- 效率：手动登录 1000 台服务器执行更新，耗时极长
- 一致性：只要其中几台因网络抖动或配置微差导致更新失败，整个系统就会陷入版本不一致的混乱

**DevOps 解决方案：**
- 自动化运维流程
- 基础设施即代码（IaC）
- 持续监控与反馈

---

### 4. 持续交付（Continuous Delivery, CD）

**核心：建立自动化流水线**

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  提交       │ -> │  自动测试   │ -> │  自动构建   │ -> │  自动部署   │
│  (Commit)   │    │ (Testing)   │    │   (Build)   │    │(Deployment) │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**流水线各阶段：**

| 阶段 | 说明 |
|------|------|
| 提交（Commit） | 开发者上传代码改动 |
| 自动测试（Automated Testing） | 系统自动运行成千上万个测试，确保新代码没有破坏原有功能 |
| 自动构建（Automated Build） | 测试通过后，系统自动把代码打包成一个新的容器镜像 |
| 自动部署（Automated Deployment） | 系统按照预设策略（如：一次更新 10%，观察没问题再继续），自动把新镜像推送到容器中 |

**Fail-Fast（快速失败）机制的好处：**

| 好处 | 说明 |
|------|------|
| 防止污染 | 坏代码不会进入生产环境，避免影响真实用户 |
| 快速反馈 | 开发者能立刻收到报错通知，在记忆最清晰时修复问题 |
| 降低成本 | 在流水线早期修复问题的代价，远小于在系统崩溃后进行抢修 |

## 三、Kubernetes（K8s）—— 云原生的"大脑"

### 核心定位

> 如果容器是集装箱，那么 Kubernetes 就是港口上的全自动调度系统

**K8s 职责：** 在拥有成千上万个容器的复杂系统中，确保每个容器都能在正确的时间、正确的地点运行

---

### 三大核心能力

#### 1. 自我修复（Self-healing）

**工作机制：**
- 如果一个容器"掉线"或崩溃了，K8s 会立刻发现并自动重启一个新的来替代它

**思考题：**
> 如果你告诉 K8s："我要 3 个支付服务的实例一直运行"，此时突然有一台服务器坏了，导致其中 1 个容器消失了，K8s 会发现"实际状态"（2 个）不等于"期望状态"（3 个），它会怎么做？

**答案：** K8s 会立刻在集群中寻找有空余资源的服务器，并自动启动一个新的容器，以恢复到你要求的 3 个状态

---

#### 2. 自动扩缩容（Auto-scaling）

**HPA（Horizontal Pod Autoscaler）工作原理：**

```
设定目标：当 CPU 使用率超过 60% 时，自动增加实例
    ↓
K8s 监控到指标超过阈值
    ↓
自动启动成百上千个新的容器实例（Pods）来分担压力
    ↓
当活动结束后流量下降，自动删除多余的容器
```

**实战价值：**
- 流量暴增时：自动扩容，无需人工干预
- 流量下降时：自动缩容，节省成本

---

#### 3. 服务发现与负载均衡（Service Discovery & Load Balancing）

**功能：**
- 自动给容器分配 IP
- 在多个容器之间平均分配流量
- 防止某个容器被"累死"

---

### 核心概念：声明式配置（Declarative Configuration）

**理解 K8s 的关键：**

> 你不需要命令它"去启动一个容器"，而是告诉它一个**"期望状态"（Desired State）**

**示例：**
```yaml
# 你告诉 K8s：
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
spec:
  replicas: 3  # "我要 3 个实例一直运行"
```

**K8s 的工作方式：**
- 你定义"终点"（我要 3 个）
- K8s 负责"路径"（怎么维持 3 个）
- K8s 持续监控实际状态，确保与期望状态一致

## 四、12-Factor App（十二要素应用）

### 核心原则

#### 1. 配置（Config）

**原则：** 不要把数据库密码硬编码在代码里，要通过"环境变量"注入

**好处：**
- 一份代码可以不经修改就在开发、测试、生产环境运行
- 安全性提升：敏感信息不进入代码仓库

**示例：**
```bash
# 传统方式（不推荐）
DATABASE_URL = "mysql://admin:password123@localhost:3306/mydb"

# 12-Factor 方式（推荐）
DATABASE_URL = os.getenv("DATABASE_URL")
# 通过环境变量注入：export DATABASE_URL="mysql://..."
```

---

#### 2. 无状态（Statelessness）

**原则：** 应用进程不应该在本地存储数据

**比喻：** 就像共享单车一样，你骑完即走，不留痕迹。数据应该存在专门的数据库里

**思考题：**
> 如果你的应用是"有状态的"（比如它把用户上传的文件存在了自己那个小容器的硬盘里），当 K8s 因为故障把这个容器关掉并在另一台机器重启时，用户上传的文件会发生什么？

**答案：** 数据会丢失！因为容器是"临时性"的（Ephemeral），它们随时可能因为故障、自动缩容或版本更新而被替换

**解决方案：**

| 方案 | 说明 |
|------|------|
| 分离（Separation） | 把"计算"（运行代码的容器）和"存储"（数据库或云存储）分开 |
| 持久化（Persistence） | 使用 K8s Persistent Volumes (PV)，像给容器插了个"外接硬盘"，即使容器坏了换成新的，数据也还在 |

---

#### 3. 其他关键要素

| 要素 | 说明 |
|------|------|
| 代码库（Codebase） | 一份代码库对应多个部署（开发、测试、生产） |
| 依赖（Dependencies） | 显式声明所有依赖，不隐式依赖系统工具 |
| 后端服务（Backing Services） | 数据库、缓存等作为附加资源，不绑定到应用 |
| 构建、发布、运行（Build, Release, Run） | 严格分离构建和运行阶段 |
| 进程（Processes） | 应用以一个或多个无状态进程运行 |
| 端口绑定（Port Binding） | 通过端口绑定提供服务，不依赖外部服务器 |
| 并发（Concurrency） | 通过进程模型处理并发，而非操作系统线程 |
| 易处置（Disposability） | 快速启动和优雅关闭，实现弹性扩缩容 |
| 开发/生产环境一致 | 保持开发、预发布、生产环境尽可能相同 |
| 日志（Logs） | 将日志视为事件流，不管理日志文件 |
| 管理进程（Admin Processes） | 管理任务作为一次性进程运行 |

## 五、实战场景：电商大促流量暴增 50 倍

### 场景设定

> 你是一家大型电商平台的首席架构师。半小时后，一年一度的"大促活动"就要开始了。根据预测，届时你的系统访问量将瞬间激增**50 倍**。

### 应对方案

#### 1. 微服务层面：隔离与保护

> 微服务隔离策略（熔断、限流、独立扩容）详见 `001-微服务/01-微服务核心知识点.md` 第三章。

---

#### 2. Kubernetes 层面：自动扩缩容

**问题：** 面对这 50 倍的流量，K8s 的哪项功能可以帮你自动增加服务器上的应用实例？

**答案：HPA（Horizontal Pod Autoscaler）**

**配置示例：**
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: payment-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: payment-service
  minReplicas: 3
  maxReplicas: 100
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60  # CPU 使用率超过 60% 时扩容
```

**执行流程：**
1. Prometheus 监控到请求延迟（Latency）上升
2. HPA 检测到指标超过阈值，向 Deployment 发出扩容指令
3. K8s 迅速调度新 Pod，并通过 Service 自动实现流量的负载均衡

---

#### 3. CI/CD 层面：紧急修复

**问题：** 如果在活动期间，你必须修复一个紧急漏洞（Bug），CI/CD 流水线如何能让你在几分钟内把修复补丁发布给全球用户？

**解决方案：**

```
代码修复 -> CI（持续集成）跑测试 -> CD（持续部署）生成新镜像 -> 推送到 K8s
```

**优势：**
- 自动化流程让你不需要手动去 1000 个容器里改代码
- K8s 执行滚动更新（Rolling Update），逐个替换旧容器
- 确保在修复 bug 的过程中，服务依然是在线可用的

---

### 遇到的难题与解决方案

#### 问题：数据库连接池爆满

**场景：** 虽然 K8s 成功扩容了 100 个容器，但数据库却因为连接数过多崩溃了

**解决方案：增加缓存层（Redis）**

**缓存策略选择：**

| 策略 | 说明 |
|------|------|
| 旁路缓存（Cache Aside） | 先读缓存，未命中再读库并回写 |
| 缓存预热 | 活动前将热点数据（库存、商品信息）提前加载到 Redis |
| 防雪崩 | 通过 TTL（生存时间）或随机过期时间防止缓存雪崩 |

**分布式锁处理并发竞争：**
> 在 50 倍流量下，如果多个请求同时修改同一件商品的库存，可能会导致数据超卖。我们利用 Redis 的 SETNX 实现了分布式锁，确保在高并发下数据的一致性。

**性能提升量化：**
- 数据库 IOPS 降低了 70%
- 系统响应时间（RT）从 500ms 降低到 50ms

## 六、云原生数据库核心特性

### 存储计算分离架构

| 特性 | 架构师视角 | 实战描述 |
|------|-----------|---------|
| 存储计算分离 | 计算节点可根据流量秒级扩容，存储按需自动增长 | "在 50 倍流量冲击下，我们快速增加了只读节点，配合 Redis 缓存分担读压力" |
| Serverless 弹性 | 自动应对峰谷流量，无需手动预估配置 | "利用 Redis 缓存高频热点数据，减少云数据库 Serverless 频繁扩容的延迟" |
| 高可用与自愈 | 多副本强一致性，节点故障秒级切换 | "当云数据库发生主备切换时，Redis 作为缓冲层确保了前端业务的短暂可用" |

### 选型依据

| 业务需求 | 技术选型 |
|---------|---------|
| 订单数据量已达数 TB，传统扩容方式停机时间过长 | 选择云原生数据库（如阿里云 PolarDB 或华为云 GaussDB） |
| 需要分钟级增加只读节点 | 看重其共享存储架构和秒级扩容能力 |

### 落地难点与应对

#### 难点 1：缓存一致性

**问题：** Redis 缓存与云原生数据库在极端高并发下的数据不一致

**对策：**
- 采用 Canal 监听数据库 Binlog 异步更新缓存
- 配合 Redis 分布式锁处理库存扣减

#### 难点 2：SQL 慢查询性能波动

**问题：** 多节点并发下，复杂 SQL 可能拖累共享存储

**对策：**
- 利用云原生数据库的并行查询（Parallel Query）技术
- 将长事务进行拆分

#### 难点 3：数据库连接数爆炸

**对策：**
- 使用数据库代理（Database Proxy）进行连接池管理
- 实施连接复用，减少直接连接数据库的开销

## 十、高并发架构设计模式

### 1. 缓存策略模式

#### 旁路缓存（Cache Aside）

```
┌─────────────────────────────────────────────────────────────┐
│                    Cache Aside 模式流程                      │
│                                                             │
│   读请求：                                                  │
│   应用 → 查询缓存 → 命中？→ 是 → 返回数据                   │
│                    ↓ 否                                     │
│         查询数据库 → 写入缓存 → 返回数据                    │
│                                                             │
│   写请求：                                                  │
│   应用 → 更新数据库 → 删除缓存 (删除而非更新)                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**关键点：**
- 写操作时**删除缓存**而非更新缓存，避免并发写导致的不一致
- 缓存设置合理的 TTL，防止数据永久过期

#### 缓存预热（Cache Warm-up）

**适用场景：** 大促活动前、新系统上线

**实施步骤：**
1. 活动前 1 小时，通过离线任务将热点数据加载至 Redis
2. 按照热度分级：Top 100/Top 1000/Top 10000
3. 设置不同的过期时间：热点数据永不过期 + 异步刷新

#### 缓存雪崩防护

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 缓存雪崩 | 大量缓存同时过期 | 设置随机 TTL，避免集体过期 |
| 缓存击穿 | 热点 Key 过期 + 高并发 | 分布式锁 + 单线程重建缓存 |
| 缓存穿透 | 查询不存在的数据 | 布隆过滤器 + 空值缓存 |

## 十三、Serverless 架构模式（2025 下-题2 真题考点）

### 1. Serverless 核心概念

**定义：** Serverless（无服务器架构）并非没有服务器，而是开发者无需关心底层服务器的管理与运维，云平台自动完成资源分配、弹性伸缩和故障恢复。

**两大核心组成：**

| 组件 | 全称 | 职责 | 典型产品 |
|------|------|------|---------|
| FaaS | Function as a Service（函数即服务） | 运行事件触发的无状态函数，按需执行 | AWS Lambda、阿里云函数计算 |
| BaaS | Backend as a Service（后端即服务） | 提供托管的后端能力（数据库、认证、存储等） | Firebase、AWS DynamoDB |

**与传统架构的核心差异：**

```
┌─────────────────────────────────────────────────────────────┐
│              传统 vs 云原生 vs Serverless 对比               │
├──────────┬──────────────┬──────────────┬────────────────────┤
│ 维度     │   单体架构    │  微服务+K8s   │    Serverless     │
├──────────┼──────────────┼──────────────┼────────────────────┤
│ 伸缩粒度  │ 整个应用      │ Pod/容器      │ 函数级             │
│ 伸缩速度  │ 分钟级        │ 秒级~分钟级    │ 毫秒~秒级          │
│ 付费模式  │ 包年包月      │ 按实例时长      │ 按执行次数+时长    │
│ 运维成本  │ 高（全管）    │ 中（管 K8s）   │ 低（零运维）       │
│ 冷启动    │ 无           │ 无            │ 有（首请求延迟）    │
│ 供应商锁定│ 低           │ 中            │ 高               │
└──────────┴──────────────┴──────────────┴────────────────────┘
```

### 2. Serverless 核心优势

| 优势 | 说明 | 实战价值 |
|------|------|---------|
| 零运维 | 无需管理服务器、操作系统、中间件 | 团队聚焦业务逻辑，运维人力减少 80% |
| 按需付费 | 仅在函数执行时计费，空闲不收费 | 低频场景成本降低 90%+ |
| 自动弹性 | 从 0 到数千实例自动伸缩 | 无需预置容量，应对突发流量 |
| 事件驱动 | 天然与事件源（HTTP、消息队列、定时任务）集成 | 简化架构，减少胶水代码 |

### 3. Serverless 核心挑战与应对

| 挑战 | 原因 | 应对策略 |
|------|------|---------|
| **冷启动延迟** | 函数首次执行需加载运行时环境 | 预预热（Provisioned Concurrency）、保持最低实例数、减小部署包体积 |
| **供应商锁定** | 各云厂商 FaaS API 和触发器绑定 | 使用 Serverless Framework/KNative 等抽象层、函数逻辑保持无状态和可移植 |
| **调试困难** | 本地环境与云端运行时差异 | 使用 LocalStack/SAM 本地模拟、全链路追踪集成 |
| **执行时长限制** | FaaS 通常限制单次执行时长（5-15 分钟） | 长任务拆分、使用 Step Functions 编排、超时任务转容器 |
| **分布式复杂性** | 函数间通过事件通信，调试链路复杂 | 引入分布式追踪（X-Ray/SkyWalking）、事件 Schema 管理 |

### 4. Serverless 适用场景

| 场景 | 典型应用 | 为什么适合 |
|------|---------|-----------|
| API 后端 | RESTful API、Webhook 处理 | 请求驱动、波动大、按需付费划算 |
| 事件驱动处理 | 文件上传触发处理、消息队列消费 | 天然事件触发、无需常驻进程 |
| 定时任务 | 数据备份、报表生成、健康检查 | 低频短任务、无需专用服务器 |
| 数据处理管道 | 日志分析、图片压缩、ETL | 计算密集但间歇性、弹性伸缩 |
| IoT 数据接收 | 设备数据上报处理 | 高并发写入、波动大、BaaS 存储配合 |

### 5. Serverless 实战：订单处理函数

```yaml
# AWS Lambda + API Gateway 配置示例
Resources:
  OrderFunction:
    Type: AWS::Serverless::Function
    Properties:
      Handler: index.handler
      Runtime: nodejs18.x
      MemorySize: 256          # 内存配置，影响 CPU 和成本
      Timeout: 30              # 超时时间
      ProvisionedConcurrency: 10  # 预置并发，消除冷启动
      Events:
        ApiEvent:
          Type: Api
          Properties:
            Path: /orders
            Method: POST
      Policies:
        - DynamoDBCrudPolicy:
            TableName: !Ref OrderTable
```

**执行流程：**
```
用户请求 → API Gateway → 触发 Lambda 函数 → 写入 DynamoDB → 返回响应
         (路由/鉴权)     (自动扩缩容)        (BaaS 存储)      (< 100ms)
```

## 十四、负载均衡设计技术（2025 上-题2 真题考点）

### 1. 负载均衡策略分类

#### 静态负载均衡策略

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 轮询法（Round Robin） | 依次将请求分配给后端服务器 | 实现简单、公平 | 不考虑后端负载差异 | 服务器性能相同、请求负载均匀 |
| 加权轮询（Weighted RR） | 按权重比例分配请求 | 可适应不同性能的服务器 | 权重需手动调整 | 服务器性能差异大 |
| 哈希法（Hash） | 根据请求特征（IP/URL）哈希选择后端 | 同一客户端固定路由到同一后端 | 后端变化时哈希重分布 | 会话保持、缓存一致性 |

#### 动态负载均衡策略

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 最小连接数（Least Connections） | 将请求发给当前活跃连接数最少的服务器 | 自动适应负载变化 | 需要实时统计连接数 | 请求处理时长差异大 |
| 最快响应时间（Fastest Response） | 将请求发给响应最快的服务器 | 自动选择性能最优节点 | 响应时间波动时不稳定 | 对延迟敏感的场景 |
| 负载反馈（Load Feedback） | 根据服务器 CPU/内存等指标动态调整 | 全面反映服务器状态 | 实现复杂、采集有延迟 | 大规模集群精细调度 |

#### 基于场景的负载均衡策略

| 策略 | 说明 | 示例 |
|------|------|------|
| 会话保持（Session Affinity） | 同一用户的请求固定路由到同一后端 | Cookie 注入、IP Hash |
| 地理位置路由 | 根据用户地理位置路由到最近的可用区 | CDN 边缘节点、GSLB 全局负载均衡 |
| 业务类型路由 | 根据请求类型路由到不同的服务集群 | 读写分离、VIP 用户专属集群 |

### 2. 负载均衡实现方式

```
┌─────────────────────────────────────────────────────────────┐
│                  负载均衡实现方式对比                        │
├──────────────┬──────────────┬──────────────┬─────────────────┤
│ 类型         │ 产品示例      │ 层级          │ 特点           │
├──────────────┼──────────────┼──────────────┼─────────────────┤
│ 硬件负载均衡  │ F5 BIG-IP    │ L4/L7        │ 高性能、高成本   │
│ 软件负载均衡  │ Nginx/HAProxy │ L7           │ 灵活、成本低    │
│ 云负载均衡    │ AWS ALB/SLB  │ L4/L7        │ 托管、自动伸缩   │
│ K8s Ingress  │ Nginx Ingress │ L7           │ 云原生集成      │
│ Service Mesh  │ Istio/Linkerd │ L7           │ 细粒度流量治理  │
└──────────────┴──────────────┴──────────────┴─────────────────┘
```

### 3. 云原生负载均衡

**Kubernetes Service 模型：**

| 类型 | 说明 | 使用场景 |
|------|------|---------|
| ClusterIP | 集群内部 IP，仅集群内可达 | 服务间内部通信 |
| NodePort | 在每个节点上开放端口 | 外部访问测试环境 |
| LoadBalancer | 对接云厂商负载均衡器 | 生产环境外部访问 |
| ExternalName | DNS CNAME 重定向 | 外部服务代理 |

**Kubernetes Ingress：**

## 十六、事件驱动架构 EDA（2025 上-题3 真题考点）

### 1. EDA 核心概念

**定义：** 事件驱动架构（Event-Driven Architecture, EDA）是一种以**事件**为中心的架构模式，通过事件的发布与订阅机制实现组件间的松耦合通信。

**核心组件：**

| 组件 | 说明 | 示例 |
|------|------|------|
| 事件（Event） | 系统中发生的有意义的事实 | "订单已创建"、"库存已扣减" |
| 事件生产者（Producer） | 产生并发布事件的组件 | 订单服务、支付服务 |
| 事件消费者（Consumer） | 订阅并处理事件的组件 | 通知服务、物流服务、分析服务 |
| 事件通道（Channel） | 传输事件的中间件 | Kafka、RabbitMQ、EventBridge |

### 2. EDA 事件模式

#### 发布-订阅模式（Pub/Sub）

```
┌──────────────┐
│  事件生产者   │  ──发布事件──→  ┌─────────────────────┐
│  (订单服务)   │                 │    事件通道 (Kafka)   │
└──────────────┘                 └──────────┬──────────┘
                                            │ 广播
                              ┌─────────────┼─────────────┐
                              ↓             ↓             ↓
                        ┌──────────┐ ┌──────────┐ ┌──────────┐
                        │ 通知服务  │ │ 物流服务  │ │ 分析服务  │
                        └──────────┘ └──────────┘ └──────────┘
```

**特点：** 一个事件可被多个消费者订阅，天然支持系统扩展

#### 事件溯源（Event Sourcing）

**核心思想：** 不存储当前状态，而是存储所有导致状态变化的事件序列

```
传统方式：  数据库存储 "用户余额 = 100"
事件溯源：  数据库存储 [
  "开户，余额 +1000",
  "消费，余额 -200",
  "充值，余额 +500",
  "消费，余额 -200",
  ...
] → 当前余额 = 100（通过事件回放计算）
```

**优势：**
- 完整审计轨迹：可追溯任意时间点的状态
- 数据不丢失：即使数据库崩溃，可通过事件回放重建
- 支持多视图：同一事件序列可生成不同的读模型

#### CQRS（命令查询职责分离）

```
┌─────────────────────────────────────────────────────────────┐
│                    CQRS 架构                                │
│                                                             │
│   写请求 → 命令模型（Command Model）→ 写入事件存储           │
│                              ↓ 发布事件                      │
│   读请求 ← 查询模型（Query Model）← 消费事件构建读视图       │
│                                                             │
│   写模型和读模型完全分离，可独立伸缩                         │
└─────────────────────────────────────────────────────────────┘
```

**适用场景：** 读写比例悬殊、读模型需要多视图、高并发读场景

### 3. EDA 技术选型

| 中间件 | 特点 | 适用场景 |
|--------|------|---------|
| **Kafka** | 高吞吐、持久化、分区有序、支持回溯 | 大数据管道、事件溯源、日志聚合 |
| **RabbitMQ** | 灵活路由、低延迟、协议丰富（AMQP） | 业务消息队列、RPC 替代 |
| **AWS EventBridge** | 托管服务、事件总线、跨服务集成 | 云原生事件驱动、SaaS 集成 |
| **Redis Streams** | 轻量、低延迟、内存存储 | 简单事件流、实时通知 |

### 4. EDA 与微服务的结合

**优势：**
- **松耦合**：服务间通过事件通信，无需直接调用
- **异步处理**：提高系统吞吐量和响应速度
- **弹性**：事件通道作为缓冲区，削峰填谷
- **可扩展**：新服务只需订阅事件即可加入系统

**挑战与应对：**

| 挑战 | 应对 |
|------|------|
| 事件顺序性 | Kafka 分区键保证同 Key 事件有序 |
| 事件重复消费 | 消费者端实现幂等性处理 |
| 事件 Schema 变更 | 使用 Schema Registry 管理版本 |
| 调试困难 | 全链路追踪 + 事件可视化监控 |
