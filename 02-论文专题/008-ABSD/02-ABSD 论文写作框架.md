# ABSD 论文写作框架

> 软考架构师论文专题 | 适用考试：系统架构设计师（2026 年 5 月）

---

## 零、审题方法论

### 0.1 ABSD 论文常见论题类型

| 论题类型 | 典型题目 | 解题要点 |
|---------|---------|---------|
| 方法论论述 | 论基于架构的软件设计方法及其应用 | 重点论述 ABSD 六阶段，结合项目实践 |
| 架构评估 | 论软件架构评估方法及其应用 | 重点论述 ATAM 四要素，写出具体权衡过程 |
| 质量属性 | 论软件质量属性驱动的架构设计 | 重点论述质量属性场景和架构战术选择 |
| 架构风格 | 论事件驱动架构在 XX 系统中的应用 | 重点论述风格选择理由和与质量属性的映射关系 |
| 架构演进 | 论软件架构的演进方法与实践 | 重点论述架构变更的受控管理和经验教训 |

### 0.2 三步审题法

```
第一步：识别关键词
  "架构驱动" "质量属性" "评估" "权衡" → ABSD 主题
  "架构风格" "微服务" "事件驱动" → 架构设计主题
  "ATAM" "SAAM" "评估" → 架构评估主题

第二步：确定论述重点
  ABSD 主题 → 六阶段 + 质量属性场景 + ATAM 评估
  架构评估主题 → ATAM 四要素 + 具体权衡案例
  架构风格主题 → 风格选择理由 + 质量属性映射 + 实现细节

第三步：匹配项目素材
  选择与论题最契合的项目经验
  准备对应的质量属性场景和架构决策案例
```

### 0.3 ABSD 论题的特殊要求

| 要求 | 说明 |
|------|------|
| 必须提及 ABSD 六阶段 | 即使题目没有明确要求，也要在正文中显式提到六阶段 |
| 必须有质量属性场景 | 至少描述 2 个完整的质量属性场景（用六要素格式）|
| 必须有 ATAM 评估 | 至少写出 2 个敏感点、1 个权衡点、1 个风险点 |
| 必须有具体项目 | 不能只谈理论，必须结合具体项目论述 |
| 必须量化效果 | 所有成果必须有具体数字支撑 |

## 一、论文结构公式

### 1.1 标准论文结构

```
摘要 (300 字) → 背景与问题 (500 字) → 核心技术方案 (800 字) → 架构评估 (600 字) → 效果与总结 (400 字)
```

### 1.2 字数分配表

| 章节 | 字数 | 内容占比 | 论述重点 |
|------|------|---------|---------|
| 摘要 | ~300 字 | 12% | 背景 + 问题 + 方法 + 成果 |
| 项目背景 | ~500 字 | 20% | 项目概况 + 旧系统痛点 + 业务影响 |
| 架构需求 | ~400 字 | 16% | 质量属性识别 + 场景描述 |
| 架构设计 | ~600 字 | 24% | 架构风格选择 + 核心方案设计 |
| 架构评估 | ~500 字 | 20% | ATAM 评估 + 权衡分析 |
| 总结反思 | ~200 字 | 8% | 成果数据 + 不足 + 展望 |
| **合计** | **~2500 字** | **100%** | — |

### 1.3 ABSD 六阶段对应章节映射

| ABSD 阶段 | 论文章节 | 段落标题建议 |
|----------|---------|------------|
| 1. 需求获取 | 第二章 | 基于 ABSD 的架构需求获取 |
| 2. 架构设计 | 第三章 | 架构设计与风格选择 |
| 3. 文档化 | 第三章 | 架构文档化描述 |
| 4. 架构评估 | 第四章 | 架构评估与权衡分析 |
| 5. 架构实现 | 第五章 | 架构实现与关键技术 |
| 6. 架构演进 | 第五章/结尾 | 架构演进与优化 |

## 二、摘要写作模板

### 2.1 摘要公式

```
项目背景 + 面临困境 + 技术方案 + 实施效果
```

### 2.2 五步写作法

```
第一步：背景 → 行业/项目背景，点明项目规模和重要性
第二步：问题 → 系统面临的核心挑战（高并发/一致性/安全等）
第三步：方法 → 明确使用 ABSD 方法论，以质量属性为驱动
第四步：措施 → 2-3 个核心架构决策（本地消息表/国密加密/微服务等）
第五步：结果 → 量化成果数据（性能提升/可用性/零事故等）
```

### 2.3 摘要模板

```
【摘要】
本文以作者参与建设的"[项目描述]"项目为背景，探讨了基于架构的软件设计
（ABSD）方法在[系统类型]中的应用。该系统整合了[核心模块]，面临[核心挑战 1]、
[核心挑战 2]以及[核心挑战 3]等严苛要求。作为系统架构师，本人通过 ABSD 方法论，
以业务质量属性为驱动，确立了以[架构风格]为核心的[技术体系]。本文重点论述了
通过[技术方案 1]实现[目标 1]，以及应用[技术方案 2]平衡[属性 A]与[属性 B]的
权衡过程。经 ATAM 评估，系统在[峰值场景]表现卓越，有效提升了[关键指标]。
项目上线后，[量化成果 1]，[量化成果 2]，圆满完成了业务目标。
```

### 2.4 范文摘要示例（物流系统）

```
【摘要】
本文以作者参与建设的"某大型第三方物流全链路协同平台"项目为背景，深入探讨了基于
架构的软件设计（ABSD）方法在复杂分布式系统中的应用实践。针对仓储管理（WMS）与
运输管理（TMS）深度集成中的分布式一致性、系统可用性及隐私合规性等核心挑战，本文
通过 ABSD 的六个生命周期阶段，确立了以事件驱动架构（EDA）为核心的微服务体系。文
中详细论述了利用"本地消息表"保障最终一致性、引入 Sentinel 实现熔断限流，以及基于
国密算法（SM4）与盲索引技术平衡安全与性能的权衡过程。通过架构权衡分析法（ATAM）
的评估，该架构在支撑千万级峰值运单的同时，确保了系统的高可用与数据合规。项目上线
后，系统吞吐量提升 300%，响应延迟降低 40%，数据安全零事故，圆满完成了业务目标。
```

### 2.5 不同领域摘要示例

| 领域 | 项目背景 | 核心挑战 | 技术方案 | 量化成果 |
|------|---------|---------|---------|---------|
| 电商 | 大型电商平台重构 | 高并发、库存一致性 | 微服务 + 事件驱动 | TPS 提升 5 倍 |
| 金融 | 银行核心系统升级 | 安全性、交易实时性 | 分层架构 + 分布式事务 | 交易延迟 < 50ms |
| 医疗 | 医院信息系统改造 | 数据共享、隐私保护 | 服务总线 + 数据脱敏 | 跨院数据互通 |
| 物流 | 智慧物流协同平台 | 分布式一致性、隐私合规 | EDA + 本地消息表 | 吞吐量提升 300% |

> **表格要点**：摘要是论文的"门面"，必须在 300 字内涵盖背景、问题、方法、成果四个要素。不同领域的摘要结构相同，但技术方案和量化成果因项目而异。

## 三、正文核心论点

### 3.1 ABSD 六阶段论述法

| 阶段 | 论述重点 | 必备内容 |
|------|---------|---------|
| 需求获取 | 如何从业务需求中提取架构需求 | 质量属性场景表（至少 2 个完整场景）|
| 架构设计 | 为什么选择这种架构风格 | 架构风格对比分析 + 选择理由 |
| 文档化 | 如何确保干系人对架构理解一致 | 提及 4+1 视图或 UML |
| 架构评估 | 发现了哪些风险和权衡 | ATAM 四要素（至少各写 1 个）|
| 架构实现 | 如何确保实现符合架构 | 代码审计、架构约束检查 |
| 架构演进 | 架构如何适应变化 | 具体演进案例 + 经验教训 |

### 3.2 质量属性场景表格写法

论文中必须用六要素格式描述质量属性场景，参考以下表格：

| 要素 | 内容 |
|------|------|
| 刺激源 | [谁发起的？用户/攻击者/系统/管理员] |
| 刺激 | [发生了什么事？具体动作] |
| 制品 | [系统哪部分受影响？具体模块/服务] |
| 环境 | [发生时的上下文？高峰期/正常运行/过载] |
| 响应 | [系统做了什么？具体行为] |
| 响应度量 | [结果如何量化？具体指标值] |

### 3.3 ATAM 四要素写法

论文中必须包含以下四种评估要素的论述：

| 要素 | 写作模板 | 示例 |
|------|---------|------|
| 敏感点 | "[架构决策] 对 [质量属性] 影响显著，测试表明 [具体影响数据]" | "SM4 加密算法对性能影响显著，全字段加密使 PolarDB CPU 负载增加 30%" |
| 权衡点 | "我们在 [属性 A] 与 [属性 B] 之间做了权衡，选择 [方案]，理由是 [分析]" | "在安全性与性能间选择分类分级加密，仅加密敏感字段，性能损耗控制在 5%" |
| 风险点 | "[架构决策] 存在 [风险]，可能导致 [后果]" | "本地消息表在高并发下可能导致数据库 I/O 瓶颈" |
| 非风险点 | "[架构决策] 经评估是安全的，因为 [理由]" | "RocketMQ 3 副本配置对当前业务规模是安全的" |

### 3.4 核心段落写作模板

**架构风格选择段落：**

```
在进行[架构风格]选型时，本人面临了[风格 A]与[风格 B]的选择。[风格 A]具有
[优势 1]和[优势 2]的特点，但在[某方面]存在不足；[风格 B]则能更好地满足
[质量属性]需求。基于 ABSD 方法论中对[核心质量属性]的优先级分析，本人最终
选择了[选定风格]，并通过[具体措施]弥补其不足之处。
```

**架构权衡段落：**

```
在架构评估阶段，通过 ATAM（架构权衡分析法），本人发现"[架构决策]"是系统的
一个关键权衡点。该决策对[质量属性 A]和[质量属性 B]产生了相互矛盾的影响：
若采用[方案 A]，则[属性 A]得到保障，但[属性 B]会受到影响；若采用[方案 B]，
则情况相反。经过综合评估，本人选择了[最终方案]，因为[理由]。经测试验证，
该方案在满足[属性 A]的同时，将[属性 B]的影响控制在可接受范围内。
```

## 六、难点与应对策略

### 6.1 ABSD 论文常见难点

| 难点 | 表现 | 应对策略 | 论文可用表述 |
|------|------|---------|------------|
| 质量属性场景不具体 | 只写"系统要快" | 用六要素格式，写入具体数字 | "系统在 10000 TPS 下响应 < 200ms" |
| ATAM 评估空洞 | 只罗列定义无实例 | 每个要素配一个具体架构决策 | "本地消息表是权衡点：可靠性 vs 存储压力" |
| 架构选择缺乏论证 | 直接说"选择了微服务" | 写出风格对比和选择理由 | "对比单体与微服务，因 XXX 选择后者" |
| 缺乏量化数据 | "效果显著提升" | 补充优化前后的具体指标对比 | "吞吐量从 2000 TPS 提升至 12000 TPS" |
| 六阶段论述不完整 | 只写了设计和实现 | 补全需求获取和评估章节 | "在需求获取阶段，我们识别了三大质量属性" |

### 6.2 核心难点深度解析

**难点一：如何写好架构权衡？**

```
错误写法："我们选择了最终一致性，因为它更好。"
正确写法："在保证 WMS 与 TMS 数据一致性的问题上，本人面临强一致性
（分布式事务）与最终一致性（消息队列 + 补偿机制）的选择。强一致性
虽能保证瞬时数据准确，但在高并发场景下会导致系统响应极慢（测试表明
延迟增加 300%）。基于 ABSD 方法论中对可用性优先级的分析，本人最终
选择了最终一致性架构，通过引入 RocketMQ 实现解耦，并设计了自动对账
服务作为补偿。该决策使系统吞吐量提升了 300%。"
```

**难点二：如何写好质量属性场景？**

```
错误写法："系统在高并发下性能良好。"
正确写法："针对可用性质量属性，我们定义了如下场景：当消息中间件
（RocketMQ）在订单出库高峰期发生瞬时故障时（刺激），系统的消息处理
服务（制品）通过触发本地消息表持久化逻辑进入降级模式（响应），确保
消息不丢失且在 MQ 恢复后 5 分钟内完成自动补发（响应度量），数据丢失
率为 0（响应度量）。"
```

**难点三：如何写好架构演进？**

```
错误写法："后来我们优化了系统。"
正确写法："在架构演进阶段，针对初期本地消息表与业务表共用数据库
导致的 I/O 争用问题，本人提出了'冷热分离'策略：将已成功发送且超过
7 天的消息迁移至阿里云 OSS，主表仅保留待处理的热数据。这一改动将
WMS 核心表的索引维护成本降低了 40%，极大提升了高并发下的写入性能。"
```

## 七、反思与展望

### 7.1 反思方向

| 反思维度 | 示例内容 |
|---------|---------|
| 架构决策遗憾 | "初期未做日志脱敏，导致安全审计时发现敏感信息泄露风险，后通过增加脱敏中间件修复" |
| 技术选型反思 | "最初选用了 Seata 分布式事务框架，后发现性能过重，切换至本地消息表方案" |
| 团队管理反思 | "架构文档化不够充分，导致开发团队对架构约束理解不一致，增加了返工" |
| 方法论应用反思 | "需求获取阶段对可修改性重视不够，后期新增业务时扩展成本较高" |

### 7.2 展望方向

| 展望方向 | 论述要点 | 论文示例段落 |
|---------|---------|------------|
| AI 预测分析 | 从被动响应到主动预测 | "引入 Transformer 模型预测仓储节点单量，实现提前扩容和波次预合并" |
| 架构自愈 | AIOps 自动故障检测与恢复 | "基于异常检测算法，在消息表积压率达阈值时自动扩容容器实例" |
| 云原生深化 | Serverless、服务网格 | "计划将部分无状态服务迁移至 Serverless 架构，进一步降低运维成本" |
| 安全深化 | 零信任架构、隐私计算 | "探索联邦学习在物流数据共享中的应用，在不出域的前提下实现联合建模" |
