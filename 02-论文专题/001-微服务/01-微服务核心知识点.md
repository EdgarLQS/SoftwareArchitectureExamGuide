## 一、微服务概述

| 对比维度 | 单体架构（Monolithic） | 微服务架构（Microservices） |
|---------|----------------------|--------------------------|
| 结构 | 所有功能打包在一起 | 拆分成独立的小服务 |
| 伸缩 | 必须复制整个应用 | 可独立伸缩特定服务 |
| 部署 | 任何改动需重新部署整体 | 可独立部署单个服务 |
| 故障隔离 | 单点故障影响全局 | 服务间相互隔离 |

**核心优势：独立伸缩（Independent Scaling）**
- 单体架构：如果"搜索"功能访问量巨大，必须复制整个庞大的应用来应对压力，非常浪费资源
- 微服务：可以只给"搜索"服务增加更多实例，而让其他访问量小的服务（如"修改资料"）保持现状

**实战场景：支付系统紧急修复**
- 单体架构：需要停止整个系统，修复后重新部署全部功能
- 微服务架构：只需修复并重新部署支付服务，其他服务（浏览、搜索）不受影响

## 二、微服务架构深度解析

### 2.1 微服务核心特征

| 特征 | 说明 |
|------|------|
| **服务即组件** | 每个服务是独立可替换的组件，可独立部署 |
| **围绕业务能力组织** | 按业务领域而非技术分层 |
| **去中心化治理** | 技术异构、各服务独立选择技术栈 |
| **去中心化数据管理** | 每个服务拥有自己的数据存储 |
| **基础设施自动化** | CI/CD、容器化、自动化编排 |
| **容错设计** | 熔断、降级、隔离，面向失败设计 |
| **演进式架构** | 服务可独立演进，不要求全局升级 |

### 2.2 微服务 vs 单体 vs SOA

| 维度 | 单体架构 | 微服务架构 | SOA |
|------|---------|-----------|-----|
| 服务粒度 | 无服务概念 | 细粒度（单一职责） | 粗粒度（业务服务） |
| 数据管理 | 共享数据库 | 独立数据库 | 共享 ESB |
| 通信方式 | 方法调用 | REST/gRPC/消息 | SOAP/ESB |
| 部署方式 | 整体部署 | 独立部署 | 集中部署 |
| 技术栈 | 单一 | 异构 | 异构 |
| 治理方式 | 中心化管理 | 去中心化 | ESB 中心化 |
| 团队规模 | 小团队 | 多个小团队（2 Pizza） | 大团队 |
| 故障隔离 | 一损俱损 | 故障隔离 | 部分隔离 |
| 适用规模 | 小型项目 | 中大型互联网应用 | 企业级集成 |

### 2.3 DDD 领域驱动设计

| 概念 | 说明 |
|------|------|
| **领域（Domain）** | 业务范围 |
| **子域（Subdomain）** | 核心域、支撑域、通用域 |
| **限界上下文（Bounded Context）** | 领域的边界，对应一个微服务 |
| **聚合（Aggregate）** | 一组相关联的对象，作为一个整体操作 |
| **聚合根（Aggregate Root）** | 聚合的入口点，外部只能通过它访问 |
| **实体（Entity）** | 有唯一标识的对象 |
| **值对象（Value Object）** | 无唯一标识、不可变的对象 |

### 2.4 拆分原则、策略与反模式

**拆分原则：**

| 原则 | 说明 |
|------|------|
| **单一职责** | 一个服务只做一件事 |
| **高内聚低耦合** | 服务内部紧密相关，服务间依赖最小 |
| **康威定律** | 架构设计与组织沟通结构保持一致 |
| **数据自治** | 服务拥有自己的数据，不共享数据库 |
| **接口稳定** | 对外接口稳定，内部实现可自由变更 |

**拆分策略：**

| 策略 | 说明 | 示例 |
|------|------|------|
| **按业务领域** | 按业务能力拆分 | 订单服务、用户服务 |
| **按数据边界** | 按数据域拆分 | 商品数据、交易数据 |
| **按用户角色** | 按使用方拆分 | 管理端、用户端 |
| **按调用频率** | 热/冷数据分离 | 热点商品服务 |

**拆分反模式：**

| 反模式 | 问题 |
|--------|------|
| **分布式单体** | 服务间强耦合，变更需同步部署 |
| **过度拆分** | 服务过多导致运维复杂、调用链路过长 |
| **按技术拆分** | DAO 服务、Service 服务（违反业务能力原则） |
| **共享数据库** | 多个服务操作同一数据库（紧耦合） |

### 2.5 微服务通信

**同步通信：**

| 技术 | 协议 | 特点 | 适用场景 |
|------|------|------|---------|
| **REST/HTTP** | HTTP + JSON | 简单通用、生态丰富 | 对外 API、跨服务调用 |
| **gRPC** | HTTP/2 + Protobuf | 高性能、强类型、双向流 | 内部服务间通信 |
| **GraphQL** | HTTP + JSON | 按需查询、减少过度获取 | 前端数据聚合 |

**异步通信：**

| 技术 | 特点 | 适用场景 |
|------|------|---------|
| **消息队列** | 解耦、削峰、最终一致 | 事件通知、异步处理 |
| **事件驱动** | 生产者不知消费者 | 数据同步、状态变更 |

**通信模式：**

| 模式 | 说明 | 示例 |
|------|------|------|
| **请求-响应** | 客户端请求，服务端响应 | REST API |
| **通知** | 单向消息，无需响应 | 日志记录 |
| **请求-异步响应** | 请求后立即返回，后续异步通知结果 | 长耗时任务 |
| **发布-订阅** | 发布者发送事件，多个订阅者消费 | 状态变更广播 |

### 2.7 负载均衡

### 2.8 熔断降级

| 模式 | 说明 |
|------|------|

| **隔离** | 限制资源使用，防止级联故障 |

### 2.12 微服务数据管理

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **每个服务独立数据库** | 完全隔离 | 数据自治、技术异构 | 跨服务查询困难 |
| **每个服务独立 Schema** | 同实例不同库 | 部署简单 | 隔离性弱 |
| **共享数据库** | 不同服务同库 | 最简单 | 紧耦合（不推荐） |

#### CQRS + 事件溯源（Event Sourcing）

| 概念 | 说明 |
|------|------|
| **CQRS** | 命令查询职责分离，读写使用不同模型，可独立伸缩 |
| **事件溯源** | 状态变更作为事件序列持久化，而非仅存当前状态 |
| **结合优势** | 写入用事件溯源保证一致性，读取用独立模型优化查询 |

**适用场景**：读写比例悬殊、需要完整审计轨迹、多视图读模型

**事件溯源示例**：
```
传统方式：  数据库存储 "用户余额 = 100"
事件溯源：  数据库存储 [
  "开户，余额 +1000",
  "消费，余额 -200",
  "充值，余额 +500",
  "消费，余额 -200"
] → 当前余额 = 100（通过事件回放计算）
```


#### 发布策略

| 策略 | 说明 | 风险 | 适用场景 |
|------|------|------|---------|
| **蓝绿部署** | 同时运行新旧版本，切换流量 | 资源成本高 | 重大版本发布 |
| **灰度发布（金丝雀）** | 逐步将流量切换到新版本（如 10%→50%→100%） | 风险可控 | 风险控制严格场景 |
| **滚动发布** | 逐个替换旧实例 | 存在混合版本期间 | K8s 默认策略 |
| **A/B 测试** | 同时发布多个版本比较效果 | 需流量分配 | 产品效果验证 |

## 三、微服务隔离与保护实战

> 完整实战场景（电商大促流量暴增 50 倍）详见 `001-云原生/01-云原生核心知识点.md`。本节聚焦微服务层面的隔离与保护策略。

### 问题

如果流量太猛导致"支付系统"反应变慢，由于它是独立的，我们可以采取什么手段来保护它，而不至于让整个购物网站（如浏览商品、搜索）都瘫痪？

### 解决方案

| 策略 | 说明 |
|------|------|
| 独立扩容 | 给"浏览和搜索"服务增加容器实例，确保用户还能正常看商品 |
| 熔断（Circuit Breaking） | 如果支付服务压力过载，系统会自动拒绝部分请求并返回"排队中"，而不是让整个支付服务彻底崩溃（像电路里的保险丝） |
| 限流（Rate Limiting） | 设置每秒请求数上限，超出部分返回友好提示 |

## 四、分布式事务与幂等性设计

### 1. 幂等性设计（Idempotency）

**定义：** 同一请求多次执行，系统最终的处理结果都是一致的

**RequestID 生成策略：**

```
改进雪花算法（Snowflake）：
┌──────────────┬──────────────┬──────────────┐
│  时间戳 41 位  │  机器 ID 10 位  │  序列号 12 位  │
│ (69 年范围)    │ (1024 个实例)  │ (4096/毫秒)   │
└──────────────┴──────────────┴──────────────┘
```

**幂等性校验流程：**
1. 前端或网关层生成 RequestID 并附带在请求头中
2. 服务端使用 `SETNX RequestID processing 30` 尝试占位
3. 如果返回 1，说明是首次请求，继续处理业务
4. 如果返回 0，说明是重复请求，查询 Redis 中的处理状态并直接返回
5. 业务处理完成后，更新 Redis 中的状态为 success/fail

### 2. 分布式锁实现

**Redis SETNX 实现：**

```java
// 获取分布式锁
boolean locked = redis.setnx(lockKey, requestId, 30);  // 30 秒过期

if (!locked) {
    // 锁已被占用，可能是重复请求或正在处理中
    return getProcessingStatus(orderId);
}

try {
    // 执行业务逻辑
    return executeBusinessLogic();
} finally {
    // 释放锁（Lua 脚本保证原子性）
    releaseLock(lockKey, requestId);
}
```

**关键点：**
- 使用唯一 requestId 标识锁的持有者
- 释放锁时校验 requestId，防止误删他人锁
- 使用 Lua 脚本保证原子性

## 五、服务稳定性保障机制

### 1. 熔断（Circuit Breaking）

**工作原理：**
```
┌─────────────────────────────────────────────────────────────┐
│                    熔断器状态转换                            │
│                                                             │
│   关闭 (Closed) → 失败次数超过阈值 → 打开 (Open)            │
│       ↑                                    │                │
│       │                                    ↓                │
│   半开 (Half-Open) ← 冷却时间结束 ← [拒绝所有请求]           │
│       │                                                    │
│       ↓ 试探请求成功                                         │
│   关闭 (Closed) - 恢复正常                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**配置参数：**
- 失败阈值：连续失败 5 次触发熔断
- 冷却时间：30 秒后进入半开状态
- 试探请求：半开状态下允许 1 个请求通过

### 2. 限流（Rate Limiting）

**常见算法：**

| 算法 | 原理 | 适用场景 |
|------|------|---------|
| 固定窗口计数器 | 单位时间内计数 | 简单场景，存在临界突变 |
| 滑动窗口计数器 | 细分时间片滑动 | 更平滑的限流 |
| 令牌桶算法 | 固定速率产生令牌 | 允许突发流量 |
| 漏桶算法 | 固定速率流出 | 严格限制速率 |

**配置示例（令牌桶）：**
```yaml
rate_limiting:
  algorithm: token_bucket
  bucket_capacity: 1000    # 桶容量（允许突发的请求数）
  refill_rate: 100/s       # 令牌补充速率
```

### 3. 降级（Fallback）

**降级策略：**

| 降级级别 | 触发条件 | 降级动作 |
|---------|---------|---------|
| 服务降级 | 服务响应超时 | 返回缓存数据或默认值 |
| 功能降级 | 系统负载过高 | 关闭非核心功能（如推荐、评论） |
| 读服务降级 | 数据库压力过大 | 只读模式，暂停写操作 |
